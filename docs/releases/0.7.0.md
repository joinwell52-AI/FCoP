# Release 0.7.0 — Role-identity uniqueness + sequence-collision fix

**Date**: 2026-04-27
**Packages**: `fcop` 0.6.5 → 0.7.0, `fcop-mcp` 0.6.5 → 0.7.0
**Protocol bumps**: `fcop_protocol_version` 1.4.0 → 1.5.0,
`fcop_rules_version` 1.6.0 → 1.7.0
**Surface compatibility**: ADR-0003 minor-boundary rules apply — one
deprecated alias (`unbound_report`) is removed, all other tools and
resources stay shape-compatible.

## Why this release exists

Two threads converged into one release.

### Thread A — "what if both windows are ME?"

ADMIN asked: *"if we tell two agents both that they are ME, what
happens?"* The answer turned out to be uncomfortable: nothing *in
the protocol itself* stops it. Rule 4 enforces single-driver per
`thread_key`, but two agents sharing the **same role code** would
quietly fight over the same filename namespace under
`docs/agents/tasks/` / `reports/` / `issues/`, break Rule 0.b's
self-review (each session sees the other's writes as "not mine"),
and produce ambiguous routing for ADMIN. The error was inevitable
*after* ADMIN had typed the second "you are ME" — there was no
defensive layer to catch it.

ADMIN's framing was sharper than my first sketch:

> "If there are two identical MEs, then the admin is at fault. We
> need to analyze it from the files on disk, because the file is
> the protocol. If a role's report already exists, there should be
> no conflict, even temporary ones should not create fundamental
> role conflicts."

In short:

1. The on-disk ledger — not runtime locks — is the authority.
2. ADMIN is bound by the same protocol as agents and must not
   double-assign role codes.
3. There is no "temp" role state. The crack must close at the
   protocol's UNBOUND → BOUND boundary.

This release implements all three.

### Thread B — "the bug bit *us* while we were documenting it"

While logging the design discussion for thread A, `write_task`
returned `TASK-20260427-001-ADMIN-to-ME.md` — but `TASK-001` was
already in `log/tasks/` (archived earlier in the session).
Renaming the new file to `TASK-002` worked, but then `write_report`
hit the same bug and produced a colliding `REPORT-001`. We had
just inferred the existence of "the same filename twice" from the
two-ME thought experiment, and the live bug demonstrated it within
the same session.

Both bugs traced to one pattern: `write_task` / `write_report` /
`write_issue` only scanned their active directory when computing
the next sequence, ignoring `log/<type>/`. The fix is
small (one helper, three call sites) but the bug class is exactly
the architectural failure thread A predicts.

## What changed

### `fcop_protocol_version: 1.5.0` / `fcop_rules_version: 1.7.0`

#### Rule 1 — Role-identity uniqueness (`fcop-rules.mdc`)

The "Invariants across both phases" section gains four new clauses
(in both Chinese and English):

1. **Role ↔ identity is 1:1.** Any non-`ADMIN` / non-`SYSTEM` role
   code is bound to *at most one* agent at any moment. "Both
   sessions are ME" has no legal intermediate state.
2. **The on-disk ledger is the single authority on occupancy.** A
   role is *occupied* iff a file in
   `docs/agents/{tasks,reports,issues,log}/` carries it as sender /
   recipient / reporter. `fcop.json` declares "this role exists",
   never "this role is online".
3. **ADMIN is bound by this rule too.** ADMIN must not assign the
   same role code to multiple agents simultaneously. Dual of the
   existing "ADMIN cannot be assigned to an agent" clause: the
   former protects the human seat, the latter protects every AI
   seat.
4. **Refusal path.** When a Phase-2 agent receives an assignment but
   `fcop_report()` shows the role is occupied by a different
   `session_id`, it must refuse under Rule 8, drop a note in
   `.fcop/proposals/double-bind-{ts}.md`, and return ADMIN three
   options: **handoff** / **co-review** ("you are X's co-reviewer"
   — a distinct binding, read-only + second-pass write) /
   **distinct role**.

#### UNBOUND step 4 — Disk-based self-check (`fcop-protocol.mdc`)

The UNBOUND protocol gains a fourth step. Before transitioning to
BOUND, the agent cross-checks the assigned role against the new
"Role occupancy" section in `fcop_report()` output:

- `UNUSED` or `ARCHIVED` → safe to BOUND.
- `ACTIVE` with the same `session_id` → resume / safe to BOUND.
- `ACTIVE` with a different `session_id` → STOP. Refuse under
  Rule 8; do not transition.

The "ok, just pick up that task" shortcut is still valid as a BOUND
transition, but the occupancy check applies the same way.

#### `LETTER-TO-ADMIN.{zh,en}.md` — ADMIN's symmetric duty

A new section sits right after "Standard opening lines" so ADMIN
learns the constraint at the moment they would type the wrong
thing. Two dual constraints in plain language: **don't be assigned
to an agent** (you are the human), and **don't assign the same
role code to two agents at once**. Lists the three escape options
when the constraint is hit. Practical advice: if two windows must
run concurrently, use team mode with distinct roles, not two PMs.

### Code changes

#### `fcop` library

- `Project.role_occupancy()` — returns
  `tuple[RoleOccupancy, ...]`, one entry per role declared in
  `fcop.json`, plus extra "ghost" rows for role codes that appear
  in the ledger but not in the config (so ADMIN can catch stale
  team layouts). Computed from filename parses + frontmatter-only
  reads — bodies are never read, so this is safe to call from an
  UNBOUND session.
- `RoleOccupancy` dataclass — new `__all__` member of
  `fcop.models`. Fields: `role`, `open_tasks`, `open_reports`,
  `open_issues`, `archived_tasks`, `last_session_id`,
  `last_seen_at`, `status` (`UNUSED` / `ARCHIVED` / `ACTIVE`).
- **Sequence collision fix** — extracted helper
  `_existing_filenames_for_seq(*dirs)` and updated `write_task`,
  `write_report`, `write_issue` to scan **both** active and archive
  directories. All three writers now use the helper.

#### `fcop-mcp`

- `fcop_report()` UNBOUND output gains a "Role occupancy" section
  rendering the new `Project.role_occupancy()` data as a
  fixed-width table, plus a four-line agent-side guide on how to
  read the table and refuse double-binds.
- `unbound_report()` — **removed.** The 0.6.3 deprecation cycle
  promised this. References to the alias in `fcop-protocol.mdc`
  were rewritten as history.

### Tests

The 587-baseline grew to 598 passing. Net +11 tests:

- **Sequence fix regressions** (3): `test_seq_skips_archived_basename`
  in `test_project_tasks.py`, `test_project_reports.py`,
  `test_project_issues.py`. Each pre-archives a same-day file and
  asserts the next write picks the next sequence (not the
  archived one).
- **Role occupancy** (7): new file `test_role_occupancy.py` covers
  empty / freshly-initialized solo / team-init order /
  active+archived transitions / `session_id` attribution / ghost
  roles / reserved-role exclusion.
- **`fcop_report` rendering** (2): `test_fcop_report_renders_role_occupancy_section`
  (zh) and `test_fcop_report_english_role_occupancy_section`
  (en) — both assert the new section appears with all team roles
  visible and `UNUSED` shown for fresh inits.
- **Alias removal** (1, replaces 4): `test_unbound_report_alias_is_gone`
  asserts `fcop_mcp.server` no longer exports the deprecated name.
  The four 0.6.x compatibility tests for the alias were removed.

Tool / resource snapshots regenerated: `tool_surface.json` no
longer lists `unbound_report`; `fcop` public-surface snapshot now
lists `RoleOccupancy`. Both regenerations are intentional surface
changes permitted at the 0.6.x → 0.7.0 minor boundary per ADR-0003.

## What this release deliberately does NOT do

- **Does not address `ISSUE-20260427-001` (Rule 0.a.1 tripwire
  blind spot for manual git / external editors).** Deferred — the
  Rule 1 hardening landing here doesn't fix it, and bundling
  scope-different fixes would dilute review focus. Will be revisited
  in a follow-up release with its own design memo.
- **Does not introduce runtime locks.** The protocol stays
  filesystem-as-truth. No `.fcop/sessions/<role>.lock`, no shared
  state between agent processes; everything keys off file
  basenames and frontmatter.
- **Does not block writes.** The double-bind detection is a
  *protocol* obligation on the agent (refuse via Rule 8); the
  filesystem itself doesn't fail-stop. ADMIN remains the loop
  closer.
