# Release 0.6.3 — host-neutral rule distribution & `fcop_report`

- **Date**: 2026-04-26
- **PyPI (after upload)**: [`fcop 0.6.3`](https://pypi.org/project/fcop/0.6.3/) · [`fcop-mcp 0.6.3`](https://pypi.org/project/fcop-mcp/0.6.3/)
- **Governs**: [ADR-0006 Host-Neutral Rule Distribution & Upgrade](../../adr/ADR-0006-host-neutral-rule-distribution.md)
- **Stability**: ADR-0003 commitments #1 (library API) and #2 (MCP tool surface) — both surfaces are **additive only**. No rename, no required-param tightening, no resource removal.

## TL;DR

Two thrusts, both ADR-0006:

1. **`fcop-mcp` finally writes the protocol rules to disk** in a form every supported host can read — Cursor `.mdc` and `AGENTS.md` (Codex CLI) / `CLAUDE.md` (Claude Code CLI). Pre-0.6.3, `Project.init()` and `init_solo()` claimed in their docstrings to "drop the bundled rules into `.cursor/rules/`" but **didn't**. That hole is closed; new `Project.deploy_protocol_rules` is the host-neutral landing point, and the MCP server gains `redeploy_rules` so ADMIN can re-deploy when the wheel-bundled rules drift past the project-local copy.
2. **`unbound_report` → `fcop_report`** rename. The tool is the canonical session-status / init report, not just the unbound-session warning, so the name was misaligned with what users were calling. `unbound_report` stays as a deprecated alias through 0.6.x and is removed in 0.7.0 — one full ADR-0003 deprecation cycle.

## What shipped

### `fcop` 0.6.3 (library)

**Added (additive, ADR-0003 #1)**:

- `fcop.rules.get_protocol_version()` — SemVer of `fcop-protocol.mdc`, symmetric to the existing `get_rules_version()`. The two documents version independently.
- `Project.deploy_protocol_rules(force=True, archive=True)` — host-neutral redeploy of four targets to the project root:
  - `.cursor/rules/fcop-rules.mdc` (Cursor — byte-exact copy)
  - `.cursor/rules/fcop-protocol.mdc` (Cursor — byte-exact copy)
  - `AGENTS.md` (Codex CLI / generic — frontmatter stripped from `fcop-rules.mdc`)
  - `CLAUDE.md` (Claude Code CLI — frontmatter stripped from `fcop-rules.mdc`)
  Stale targets are archived under `.fcop/migrations/<ts>/rules/<original-path>` before being overwritten. Returns a `DeploymentReport` listing every file touched.
- `Project.init(deploy_rules=True)` — `deploy_rules` now wires through `deploy_protocol_rules` so a fresh project ships with the four rule targets already on disk. Existing call sites without the flag are unchanged.

**Changed**: nothing in the existing public API. Tests added; public-surface snapshot regenerated for the additive exports.

### `fcop-mcp` 0.6.3 (MCP server)

**Added (additive, ADR-0003 #2)**:

- **`fcop_report(lang="zh")`** — new canonical session-status report. Same body shape as the legacy `unbound_report`, plus a `[Versions]` / `[版本]` block:

  ```
  [Versions]
    fcop-mcp:  0.6.3
    fcop:      0.6.3
    rules:     local 1.5.0 | packaged 1.5.0 ✓
    protocol:  local 1.4.0 | packaged 1.4.0 ✓
  ```

  When the project-local rules / protocol files drift behind the wheel-bundled ones, the line flips to `OUTDATED` / `本地偏旧` and the report appends an ADMIN-facing prompt: "run `redeploy_rules` to refresh".

- **`redeploy_rules(force=True, archive=True, lang="zh")`** — ADMIN-only thin wrapper over `Project.deploy_protocol_rules`. Agents must not invoke directly; the docstring says so. The MCP surface stays symmetric with `deploy_role_templates` so ADMIN can call it from the chat box.

**Deprecated (one full minor, removal in 0.7.0)**:

- **`unbound_report`** is now a thin alias of `fcop_report`. Every call emits

  ```
  DeprecationWarning: unbound_report is deprecated; use fcop_report instead.
  This alias will be removed in fcop-mcp 0.7.0. See ADR-0006 for the rationale.
  ```

  Behaviour is otherwise identical — same output bytes, same path-resolution cascade, same lang flag. Migration is purely lexical: replace every `unbound_report` mention in your system prompts / `LETTER-TO-ADMIN.md` / docs with `fcop_report`.

### Tool surface count

- **0.6.0–0.6.2**: 24 MCP tools.
- **0.6.3**: **26** tools — the original 24 (including the now-deprecated `unbound_report` alias) plus `fcop_report` and `redeploy_rules`.

## Migration

### For end users (Cursor / Claude / Codex)

Pure upgrade, no config change required:

```bash
pip install -U "fcop==0.6.3" "fcop-mcp==0.6.3"
```

After upgrade, the **first thing** to do in any existing project:

1. Reload the MCP server in your IDE.
2. Have ADMIN call `redeploy_rules` once. This writes the four rule targets to disk; subsequent sessions see drift-free `[Versions]` blocks and every host (Cursor / Claude / Codex / raw API) gets the same protocol text.

If you want to keep using `unbound_report` for a while, you can — it still works, just emits a `DeprecationWarning` you'll see in the MCP server log. **Recommended**: do a one-shot find-and-replace `unbound_report` → `fcop_report` in your system-prompt template and your `LETTER-TO-ADMIN.md` so 0.7.0 doesn't break you.

### For `fcop` library users

Same wheel as before, plus two new public symbols. Existing code is byte-compatible:

```python
from fcop import Project
from fcop.rules import get_rules_version, get_protocol_version  # NEW

p = Project(".")
p.init(team="dev-team", deploy_rules=True)  # NEW: deploys protocol rules
report = p.deploy_protocol_rules(force=True)  # NEW: explicit re-deploy
```

## Release engineering (maintainers)

- ADR-0006 lands the `redeploy_rules` MCP tool and the `Project.deploy_protocol_rules` library method as **two halves of the same contract**: ADMIN sees the report (drift detected), ADMIN runs the tool, the library method writes the bytes. Either half without the other is incomplete.
- The deprecation cycle for `unbound_report` requires:
  1. **0.6.3** ship the alias + `DeprecationWarning` (this release).
  2. **0.6.x and 0.7.0-rc** keep the alias (per ADR-0003 §"Deprecation Cycle").
  3. **0.7.0 final** delete the alias from the tool registry. Update `tool_surface.json` snapshot at that time.
- The `tool_surface.json` snapshot was regenerated with `pytest tests/test_fcop_mcp/test_tool_surface.py --snapshot-update` and the `surface-check` CI job verified the change is announced in this CHANGELOG section, per ADR-0003 commitment #2.
- 首次为 `v0.6.3` 打 tag 时，`.github/workflows/release.yml` 里 **verify** 已修复（不再被 `_version.py` 文档行误匹配）；详见 0.6.2 release notes 的 *Release engineering* 节。
