# Release 0.6.4 — init-deposit fix + Solo template bundle + agent-install prompt

- **Date**: 2026-04-26
- **PyPI (after upload)**: [`fcop 0.6.4`](https://pypi.org/project/fcop/0.6.4/) · [`fcop-mcp 0.6.4`](https://pypi.org/project/fcop-mcp/0.6.4/)
- **Governs**: continuation of [ADR-0006 Host-Neutral Rule Distribution & Upgrade](../../adr/ADR-0006-host-neutral-rule-distribution.md); no new ADR.
- **Stability**: ADR-0003 commitments #1 (library API) and #2 (MCP tool surface) — both surfaces are **additive only**. No rename, no required-param tightening, no resource removal.

## TL;DR

A hot-fix release covering three protocol-vs-implementation gaps that
turned up while writing the 0.6.3 customer tutorial:

1. **`init_*` was lying about what it deposited.** `LETTER-TO-ADMIN.md`,
   the `workspace/` cage, and the three-layer team docs were advertised
   in docstrings and the user manual but never actually written. 0.6.4
   makes every `init_*` land its full promised set in a single call,
   pinned by `tests/test_fcop/test_init_promises.py`.
2. **Solo mode had no template bundle.** `init_solo` followed by
   `deploy_role_templates(team="solo")` raised `TeamNotFoundError`.
   0.6.4 ships `teams/_data/solo/` with the full three-layer
   documentation (`README.md` / `TEAM-ROLES.md` /
   `TEAM-OPERATING-RULES.md` / `roles/ME.md`, zh + en) and registers
   it in the team index so it loads exactly the same way the four
   existing teams do.
3. **Agents were defaulting init mode on ADMIN's behalf.** 0.6.3's
   Phase-1 contract was permissive enough that agents would call
   `init_project(team="dev-team")` before ADMIN said "use dev-team",
   which silently overwrote the intended Solo flow. 0.6.4 (a) has
   `fcop_report` Phase-1 explicitly tell the agent it must wait,
   (b) adds the prohibition to `fcop-rules.mdc` Rule 1, and (c) adds
   `Rule 0.a.1` forbidding the role-side "simple-tasks-may-run-directly"
   soft-constraint that lets agents skip `task → do → report → archive`.

Plus: a new MCP resource pair (`fcop://prompt/install` zh + en)
holding the canonical "have an agent install fcop-mcp for you"
prompt, the same text rendered in `mcp/README.md` and the root
README so customers can copy it from whichever surface they happen
to be reading.

All changes are additive per ADR-0003.

## What shipped

### `fcop` 0.6.4 (library)

**Fixed**:

- `Project.init` / `Project.init_solo` / `Project.init_custom` now
  deposit the full set in a single transaction (the 0.6.3 docstring
  promise minus the silent gaps):
  - `docs/agents/fcop.json`
  - `docs/agents/LETTER-TO-ADMIN.md` (picked from zh / en per `lang`)
  - `docs/agents/{tasks,reports,issues,shared,log}/`
  - `docs/agents/shared/TEAM-README.md` / `TEAM-ROLES.md` /
    `TEAM-OPERATING-RULES.md` / `roles/{ROLE}.md` (zh + en;
    skipped for custom teams since they have no bundled templates)
  - `workspace/README.md` (Rule 7.5 cage explainer)
  - `.cursor/rules/fcop-rules.mdc` + `.cursor/rules/fcop-protocol.mdc`
    + `AGENTS.md` + `CLAUDE.md` (when `deploy_rules=True`,
    matching the 0.6.3 default for the MCP entry tools)

**Added (additive, ADR-0003 #1)**:

- `Project.init` / `init_solo` / `init_custom` ↪ new
  `deploy_role_templates: bool` parameter. Defaults: `True` for
  preset / solo so the three-layer `shared/` docs land at init
  time; `False` for custom since custom teams have no bundled
  templates.
- `Project.init(team="solo")` now raises `ValueError` before
  touching the disk, pointing callers at `init_solo()` so the
  saved config carries `mode="solo"` (not `mode="preset",
  team="solo"`).
- `fcop.rules.get_install_prompt(lang)` — returns the canonical
  "have an agent install fcop-mcp" prompt (zh / en).
- `teams/_data/solo/` — Solo team template bundle. Three-layer
  documentation (`README.md` / `TEAM-ROLES.md` /
  `TEAM-OPERATING-RULES.md` + `roles/ME.md`), zh + en. The
  `ME.md` charter contains a **workflow hard constraint**
  section that explicitly forbids the soft-constraint pattern
  "simple tasks may run directly" — the exact 0.6.3 mis-design
  that let agents bypass `task → do → report → archive`.
- **All 17 bundled role charters** now open with the same
  workflow hard-constraint section (the role-side translation
  of Rule 0.a.1). Coverage:
  `solo/ME`, `dev-team/{PM,DEV,QA,OPS}`,
  `media-team/{PUBLISHER,COLLECTOR,WRITER,EDITOR}`,
  `mvp-team/{MARKETER,RESEARCHER,DESIGNER,BUILDER}`,
  `qa-team/{LEAD-QA,TESTER,AUTO-TESTER,PERF-TESTER}` — both
  `zh` and `en`, 34 files total. Pinned by
  `tests/test_fcop/test_role_templates.py` (36 tests).
- **`init_*` tools now auto-surface the letter in the chat
  reply.** Every `init_project` / `init_solo` /
  `create_custom_team` response now embeds the LETTER-TO-ADMIN
  intro slice (title + 0.6.4 summary + identity diagram) plus
  an explicit "paste verbatim to ADMIN" instruction. Backed by
  the new `fcop.rules.get_letter_intro(lang)` accessor (7 new
  tests pin it as a strict prefix of the full letter).
- **Install prompt drift tests** (11 new tests) lock the
  four-surface contract for the canonical agent-install
  prompt: bundled `agent-install-prompt.{zh,en}.md`,
  `fcop.rules.get_install_prompt(lang)`,
  `fcop://prompt/install` MCP resource, and the verbatim embed
  in `mcp/README.md` (PyPI-visible) all stay byte-for-byte
  aligned, and the safety clauses (preserve existing
  `mcpServers`, 30s–1min cooldown, do-not-auto-init) can't be
  silently dropped by a future copy edit.

### `fcop-mcp` 0.6.4 (MCP server)

**Added (additive, ADR-0003 #2)**:

- New MCP resources `fcop://prompt/install` and
  `fcop://prompt/install/en`. Total resource count: **10 → 12**.
- `init_solo` / `init_project` / `create_custom_team` all gain a
  new `force: bool` parameter (default `False`). Setting `True`
  overwrites an already-initialized project; previous artifacts
  are archived under `.fcop/migrations/<timestamp>/`. This is
  the supported way for ADMIN to switch teams (e.g. solo →
  dev-team) without manually wiping the project.

**Changed**:

- `fcop_report` Phase-1 output now (a) explicitly tells the
  agent it MUST NOT pick an init mode on ADMIN's behalf, and
  (b) points ADMIN at `fcop://letter/zh|en` for the manual when
  the three-way choice is unfamiliar.

### Protocol rules

**`fcop-rules.mdc` 1.5.0 → 1.6.0**:

- New **Rule 0.a.1** sub-section: workflow hard constraint.
  Every piece of work must follow `task → do → report → archive`.
  Role documents are forbidden from softening this with "simple
  tasks may run directly" or equivalents — that pattern itself
  becomes a Rule 0.a violation, escalatable under Rule 8.
- **Rule 1 Phase 1** rewritten:
  - lists the full set of files an `init_*` tool promises to
    deposit, so a partial deposit becomes a recognisable bug
    rather than a silent regression;
  - explicitly forbids agents from defaulting to `dev-team` /
    `solo` / `custom` on ADMIN's behalf;
  - adds explicit reference to `qa-team` (already shipped, just
    not previously listed in the rule body).

## How to upgrade

If you are already on 0.6.x, the upgrade is one line per package
inside the same MCP venv:

```bash
"$VENV/bin/pip" install -U "fcop" "fcop-mcp"
```

Then in a fresh chat, ask the agent to refresh the local
protocol rules:

> "Run `redeploy_rules()`, then run `fcop_report()` and show me
> the `[Versions]` block."

The version block should now read `1.6.0` for `fcop_rules_version`
(check that line; if it still says `1.5.0`, the wheel didn't pick
up the bump and you need to reinstall in a clean venv per
[`mcp/README.md`](../../mcp/README.md)).

## Compatibility & deprecations

- No deprecations introduced in this release.
- The `unbound_report` → `fcop_report` deprecation cycle started
  in 0.6.3 continues unchanged; `unbound_report` is removed in
  0.7.0.

## What's intentionally **not** in this release

- ADR-0006 already governs rule-file distribution; no new ADR
  was needed for this release.
