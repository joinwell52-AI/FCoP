## 本次发布概览

v1.5.0 是继 v1.4.0 之后的第一个 minor 版本，聚焦于**协议文档同步**和**自动化合规扫描能力扩展**：

1. **P1（文档层）**：将 v1.0~v1.4 的协议能力（REVIEW envelope / `risk_level` / `fcop_audit` / `supersedes:`）写入 84 份已部署的角色和团队文档，使接手项目的 Agent 无需阅读 CHANGELOG 即可感知当前协议能力。
2. **P2（扫描层）**：新增 `_scan_outdated_role_docs()` 扫描方法，在 `fcop_audit(scope=upgrade/takeover)` 时自动检测已部署角色文档的协议版本滞后（RULE_DOC_DRIFT），并给出整改建议。
3. **CI 修复**：Protocol Version Log 补录 + `drop_suggestion` 豁免。
4. **essays 07-10**：4 篇来自 codeflow 项目 dogfood 的现场报告和设计哲学文章。

## 变更详情

### P1 — 角色/团队文档协议同步（58 个文件）

**背景**：v1.4.0 发布后，通过 `deploy_role_templates()` 部署到 `fcop/shared/roles/` 和团队目录的文档仍停留在 v1.1 内容层。新接手项目的 Agent 读取旧文档，不会感知 v1.0~v1.4 引入的新协议能力，造成**协议能力认知漂移（RULE_DOC_DRIFT）**。

**升级动作**：重新部署角色模板（见"升级指南"节）

**变更文件**：

#### 非 leader 角色文档 × 20 份（10 个角色 × 中英）
`DEV` / `QA` / `OPS` / `WRITER` / `TESTER` 等非 leader 角色文档，每份新增：

```markdown
## v1.0 ~ v1.4 协议更新速查

### REVIEW envelope（v1.0）
### risk_level 字段（v1.1）
### fcop_audit 与 INSPECTION（v1.3）
### supersedes 字段（v1.4）
```

#### 团队 TEAM-ROLES × 10 份（5 团队 × 中英）
新增"协议演进说明（v1.0 ~ v1.4）"版本表格：

| 版本 | 变更 | 影响角色 |
|---|---|---|
| v1.0 | REVIEW envelope | leader / 全部 |
| v1.1 | `risk_level` 字段 | leader 派单 |
| v1.3 | `fcop_audit()` + GAL | leader / ADMIN |
| v1.4 | `supersedes:` 字段 + write-side 绑定 | 全部 |

#### 团队 TEAM-OPERATING-RULES × 10 份（5 团队 × 中英）
新增"协议演进补记"节，涵盖高风险任务审批 / fcop_audit 整改处理 / write-side 绑定规则。

#### 团队 README × 10 份（5 团队 × 中英）
新增"工具速查链接"表格（`fcop_audit` / `fcop_report` / `fcop_list_alerts` 使用场景）。

#### 其他文档
- `docs/getting-started.{zh,en}.md`：新增"fcop_audit 三场景体检"节
- `src/fcop/rules/_data/agent-install-prompt.{zh,en}.md`：装完即跑 `fcop_audit()` 提示

### P2 — `_scan_outdated_role_docs()` 新增扫描

**新增方法**：`Project._scan_outdated_role_docs()`

**触发范围**：`scope=upgrade` 和 `scope=takeover`

**检测逻辑**：

```python
def _scan_outdated_role_docs(self) -> list[Violation]:
    # 1. 读取已安装 fcop 包版本（major.minor）
    # 2. Glob fcop/shared/roles/*.md
    # 3. 对每个文件：用正则 r"v(\d+)\.(\d+)" 提取最高版本号
    #    - 无版本引用 → 记入 outdated
    #    - 版本 gap > 1 minor → 记入 outdated
    # 4. outdated 非空 → Violation(severity=P1, rule="RULE_DOC_DRIFT")
```

**违规类型**：`RULE_DOC_DRIFT`（角色文档协议漂移）

**整改建议**：`deploy_role_templates(force=True)`

**ADR-0032 更新**：§4.7 新增完整方法规格文档

**新增测试**（`tests/test_fcop/test_audit.py`）：

| 测试用例 | 场景 |
|---|---|
| `test_scan_outdated_role_docs_no_roles_dir` | 无 `shared/roles/` 目录 → 无违规 |
| `test_scan_outdated_role_docs_no_version_reference` | 文档无版本标记 → P1 违规 |
| `test_scan_outdated_role_docs_far_behind` | 仅引用 v1.0（落后 > 1 minor）→ P1 违规 |
| `test_scan_outdated_role_docs_current` | 引用当前或接近当前版本 → 无违规 |

### fix(ci) — Protocol Version Log + drop_suggestion 豁免

**问题 1**：`fcop-protocol.mdc` 正文 Protocol Version Log 缺 `v2.2` 条目，CI 测试
`test_fcop_protocol_frontmatter_matches_body_changelog` 失败。

**修复**：补录 `v2.2`（2026-05-12）条目，涵盖 `supersedes:` 字段和 GATE Design Pitfalls。

**问题 2**：`drop_suggestion` 是轻量级预初始化操作（在项目未完成 `init_project` 前
向 `.fcop/proposals/` 写建议文件），但 v1.4.0 新增的 S1 守门要求 `fcop.json` 存在，
导致 `test_drop_suggestion` CI 失败。

**修复**：`drop_suggestion` 加入 `_INIT_TOOLS` 豁免集合，与 `init_project` / `init_solo` / `create_custom_team` 同列。

## 升级指南

### 从 v1.4.x 升级

```bash
pip install -U fcop fcop-mcp
```

**重要**：升级后需重新部署角色模板，使已有项目的 `fcop/shared/roles/` 获得 P1 协议同步内容：

```python
from fcop import Project
proj = Project(".")
proj.deploy_role_templates(force=True)
```

或通过 MCP 工具：

```
deploy_role_templates(force=True)
```

**验证**：

```python
proj.audit(scope="upgrade")  # 检查 RULE_DOC_DRIFT 等合规缺口
```

运行 `fcop_audit(scope="upgrade")` 后，若所有角色文档已是最新，扫描结果应为空（无 P1 违规）。

### 从 v1.3.x 或更早版本升级

请先参考 [v1.4.0 升级指南](./1.4.0.md)，再按 v1.4.x → v1.5.0 步骤操作。
