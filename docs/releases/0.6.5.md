# Release 0.6.5 — Rule 0.a.1 tool-layer tripwires (text → behavior)

- **Date**: 2026-04-27
- **PyPI (after upload)**: [`fcop 0.6.5`](https://pypi.org/project/fcop/0.6.5/) · [`fcop-mcp 0.6.5`](https://pypi.org/project/fcop-mcp/0.6.5/)
- **Governs**: closes the gap left open by 0.6.4 — no new ADR. Continues to honor [ADR-0003 (additive-only public surfaces)](../../adr/ADR-0003-pre-1.0-stability-charter.md) and [ADR-0006 (host-neutral rule distribution)](../../adr/ADR-0006-host-neutral-rule-distribution.md).
- **Stability**: tool signatures unchanged, MCP tool-surface snapshot unchanged, no new public Python API, `public_surface.json` snapshot unchanged. Pure additive return-text and behavior nudges.

---

## TL;DR

0.6.4 deployed the `task → do → report → archive` hard constraint as
**text** in 17 role charters and `fcop-rules.mdc Rule 0.a.1`. The very
first real-world solo test of 0.6.4 (`init_solo` → ADMIN: "做个俄罗斯
方块") immediately exposed that the constraint never actually **bit**:
the agent could recite the four steps perfectly when asked but skipped
Step 1 (`write_task`) on the live request, jumped straight into
`new_workspace` + code, and only acknowledged the violation after
ADMIN called it out. The agent's own post-mortem in chat:

> "是我没有把刚建立的协议作为当前工作流的硬约束执行到底。"
>
> ("I didn't execute the just-established protocol as a hard
> constraint on the current workflow.")

That is the diagnosis: **rules in long-context system prompts get
out-prioritized by the user's task goal at the moment of action**.
You can't fix this by writing the rule one more time in one more
markdown file — the rule has to live where the agent's hand actually
moves: the tool return path.

0.6.5 plants two soft tripwires at the two structural choke points
between "ADMIN said something in chat" and "agent has landed
artifacts on disk":

1. **`new_workspace` Rule 0.a.1 reminder.** The single moment an
   agent commits to dropping artifacts. If no open `TASK-*.md`
   mentions the slug it's about to create, the tool prepends a
   bilingual reminder showing the `write_task(...)` call template
   the agent should have done first. Workspace creation still
   succeeds (additive, non-blocking).
2. **`fcop_report` four-step template.** The status report every
   bound agent reads on session start (and re-reads when it
   self-checks state) now ends with an explicit step-by-step
   reference: `write_task` → `new_workspace` → `write_report` →
   `archive_task`, plus the "skipping Step 1 or Step 3 violates
   Rule 0.a.1" callout. Both `zh` and `en`.

Both tripwires are additive (no signature changes, no new
parameters, no broken legitimate flows) and target the exact moment
LLMs prioritize rules text against task goals.

---

## Why 0.6.4's text-only fix wasn't enough

0.6.4 hardened the rule in three textual places:

| Surface | 0.6.4 hardness |
|---|---|
| `solo/roles/ME.md` (and 33 other charter files) | Rule 0.a.1 stated, with explicit "no `simple = skip`" + 0.6.3 failure log |
| `fcop-rules.mdc` Rule 0.a.1 | Stated, with the four tool names |
| LETTER-TO-ADMIN intro (auto-spliced into `init_*` reply) | Mentioned at orientation time |

All three are correct. The agent in the failed test had read all of
them. **Reading is not the bottleneck.** The bottleneck is that, at
the precise moment ADMIN says "做个游戏", the LLM's next-token
distribution is dominated by the request, not by the rules in the
system prompt that have been sitting there since session start.

The fix has to be *temporally local* to the bad action — the
reminder must arrive in the *output of the very tool* the agent is
about to call too soon.

## Fix 1 — `new_workspace` Rule 0.a.1 tripwire

### Behavior

```text
> tool: new_workspace(slug="block-starfall", title="Tetris game")

⚠️  FCoP Rule 0.a.1 check: no open TASK-*.md mentions slug `block-starfall`.
    FCoP Rule 0.a.1 检查：当前没有任何开放 TASK-*.md 提及 slug `block-starfall`。

If you just got a request from ADMIN, Step 1 is **write_task** —
BEFORE editing any file. The 4-step cycle has no "simple = skip" exception.
如果你刚收到 ADMIN 的需求，第 1 步应当是 **write_task** ——
在动手写任何文件之前。4 步循环没有「简单任务可跳过」这种例外。

  write_task(
      sender="ADMIN", recipient="<your role>", priority="P2",
      subject="<one-line goal>",
      body="<rephrase the request + scope + acceptance criteria>",
  )

Then drop artifacts into workspace/block-starfall/, then write_report when done.
随后把产物落到 workspace/block-starfall/，完成时再调 write_report 收尾。

(Workspace was created — this is a reminder, not a block.
笼子已建好——这是提醒，不阻断创建。)
---
Created workspace: `workspace/block-starfall/`
  slug: block-starfall
  title: Tetris game
  ...
```

### Detection logic

`_recent_task_mentions_slug(project, slug)` scans `docs/agents/tasks/`
(open status, capped at 200 most-recent entries) and returns `True`
iff any task has the slug as a case-insensitive substring of:

- frontmatter `subject`
- frontmatter `references` entries (e.g. `workspace/<slug>`)
- body text

If `True`, no warning is emitted. If `False` (or any IO/parse error
occurs — swallowed, never raised), the bilingual block is prepended.

### Why a soft tripwire, not a hard block

A hard block (`new_workspace` refuses to run unless task exists) was
considered and rejected:

- Backwards-incompatible: 0.6.x exists, users have flows that don't
  always match a slug to a task.
- Some legitimate uses don't fit (offline experiments, playground
  workspaces, slug names that genuinely have nothing to do with the
  task subject).
- ADR-0003 commits to additive-only changes in 0.6.x.

A soft reminder is enough: agents in practice read tool output
immediately and re-orient. Future versions can tighten if needed.

## Fix 2 — `fcop_report` four-step template

### Behavior (zh, abbreviated)

```text
=== FCoP UNBOUND 报告 ===
项目 / project: D:\hello-fcop  (来源: ...)
团队 / team: solo  负责人 / leader: ME  lang: zh
角色 / roles: ME

[Versions / 版本]
  ...

任务 tasks: 0 进行中, 0 已归档
报告 reports: 0  问题 issues: 0
最近活动 / recent activity:
  (none yet)

ADMIN 请用这句话给本会话分配角色：
  「你是 <ROLE>，在 <team>，线程 <thread_key>（可选）」

在此之前，本会话禁止：
  - 读取任务正文（只看元数据）
  - 写入任何文件
  - 从上下文自认角色
  - 派发后续任务

ADMIN 把你绑定到角色之后，**每一条**新需求都要走 Rule 0.a.1 4 步循环——
没有「简单任务可跳过」这种例外：
  第 1 步  write_task(sender="ADMIN", recipient="<你的角色>", ...)
           ← 在动手写任何文件之前
  第 2 步  做事（new_workspace + 在 workspace/<slug>/ 下落产物）
  第 3 步  write_report(task_id="TASK-...", reporter="<你的角色>", ...)
           ← 在聊天里说「做完了」之前
  第 4 步  archive_task("TASK-...")（ADMIN 验收后）
跳过第 1 步或第 3 步即违反 Rule 0.a.1。
```

The English variant carries the same template with the same four
step labels and the same "no `simple = skip`" callout phrasing as
`fcop-rules.mdc` Rule 0.a.1, so quote-matching tools and tests can
pin both surfaces against drift.

### Why both `fcop_report` *and* `new_workspace`

Different shots at the same agent at different moments:

- `fcop_report` runs at session start and during self-checks → lays
  down the four-step template before any user request arrives.
- `new_workspace` runs at the moment of artifact commitment → catches
  the case where the template was forgotten between session start
  and the user's request landing.

Both together make the rule fire at multiple temporally-local points.

## Compatibility / risk

- ADR-0003 §1 (library API): no change. `fcop` library does not
  expose new symbols. `fcop.rules.get_letter_intro` /
  `get_install_prompt` (added in 0.6.4) unchanged.
- ADR-0003 §2 (MCP tool surface): no change. Tool signatures and
  `tool_surface.json` snapshot are byte-identical to 0.6.4.
- ADR-0006 (host-neutral rules): no change.
- Performance: `_recent_task_mentions_slug` reads up to 200 most-
  recent open tasks per `new_workspace` call. On a healthy project
  (≤ a few dozen open tasks) this is < 50 ms. Errors are swallowed.
