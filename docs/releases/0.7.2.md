# 0.7.2 — Metadata patch & "third strike" guard rail

> **Released**: 2026-04-27 · **Same day as 0.7.1**
>
> **Scope**: One-line bug fix in a bundled `.mdc` frontmatter, plus
> three regression tests that make the *class* of bug behind it
> structurally unshippable. **No behaviour change, no API change, no
> protocol bump.** Released because PyPI cannot serve a corrected
> `0.7.1` wheel without a new version number.
>
> **No PyPI yank** for `fcop-mcp 0.7.0` *or* `0.7.1`; same rationale
> as documented in [`0.7.1.md`](./0.7.1.md). 0.7.2 is the higher
> version on the index, so any `pip install -U` / `uvx --refresh`
> automatically picks it up.

## What was wrong with 0.7.1

`fcop_report()` against a fresh 0.7.1 install printed:

```
[版本 / Versions]
  fcop-mcp:  0.7.1
  fcop:      0.7.1
  rules:     （项目本地未部署）| 包内 1.7.0    ← stale
  protocol:  （项目本地未部署）| 包内 1.6.0    ← correct
```

But the bundled `fcop-rules.mdc` *file* contained:

- `fcop_rules_version: 1.7.0` in the YAML frontmatter (line 4)
- `**1.8.0 changes / 1.8.0 变更**:` in the body changelog (line 563)
- The rule text actually shipped at the **1.8.0 level** (Rule 1
  sub-agent clause, Rule 5 `AMEND-*` / `-v2` removal, Rule 0.a.1
  applicability clarification — all introduced in the 0.7.1 protocol
  bump and described inside this same file).

Substantively the rules are 1.8.0. The frontmatter is 1.7.0. Two
metadata copies of the same number drifted apart in a single PR.

## Why this matters more than a typo

This is the **third "multi-line edit, one edit dropped" bug** in the
0.7.x cycle in 24 hours:

| # | Where | What was edited | What was missed | Symptom |
|---|---|---|---|---|
| `ISSUE-20260427-006` | `mcp/pyproject.toml` | description, version | `fcop>=0.6,<0.7` dep pin | `uvx fcop-mcp 0.7.0` → `ImportError` |
| REPORT-005 / 006 | yank decision | release notes intro | 4 yank-implies-blocked sentences | doc said "needs yank" after we decided not to yank |
| **ISSUE-20260427-007** | `fcop-rules.mdc` | body changelog, rules text | frontmatter `fcop_rules_version` | `fcop_report()` lies about rules version |

Each one was caught by **a human reading the released artifact**.
None were caught by the test suite, because none of the tests asked
*"are the two copies of this number equal?"*. That is the gap 0.7.2
closes.

## What 0.7.2 changes

### 1. The actual fix (one line)

`src/fcop/rules/_data/fcop-rules.mdc` line 4:

```diff
- fcop_rules_version: 1.7.0
+ fcop_rules_version: 1.8.0
```

That is the entire bug fix. The file body, the rule text, and the
deployed `.cursor/rules/fcop-rules.mdc` (after `redeploy_rules`) now
all agree.

### 2. The guard rail (three new tests)

`tests/test_fcop/test_rules_metadata_consistency.py`:

- `test_fcop_rules_frontmatter_matches_body_changelog`: parses
  `fcop-rules.mdc`, extracts `fcop_rules_version` from the YAML
  frontmatter, scans the body for all `**X.Y.Z changes ...**`
  headings, takes the maximum, and asserts equality. If a future
  edit bumps the body (because someone is documenting "what changed
  in N+1") but forgets the frontmatter — or vice versa — `pytest`
  fails before any wheel is built.
- `test_fcop_protocol_frontmatter_matches_body_changelog`: same
  contract for `fcop-protocol.mdc`, with the body convention being
  `- **vX.Y** (YYYY-MM-DD) — ...` lines under "Protocol Version
  Log".
- `test_no_stale_rules_version_anywhere_in_data`: a belt-and-braces
  combined check covering both files at once, so even a copy-pasted
  third bundled rule file would have to follow the convention.

The pre-existing `tests/test_fcop/test_pyproject_pins.py` (added in
0.7.1, defending against ISSUE-006) and these three (added now,
defending against ISSUE-007) together cover every "two copies of
the same version number" relationship the project currently relies
on:

| Pair of copies | Defended by |
|---|---|
| `fcop._version.__version__` ↔ `fcop_mcp._version.__version__` (lockstep MINOR) | `test_pyproject_pins::test_fcop_and_fcop_mcp_versions_are_aligned` |
| `fcop_mcp._version.__version__` ↔ `mcp/pyproject.toml` `fcop>=...` pin | `test_pyproject_pins::test_fcop_mcp_pin_matches_fcop_minor` |
| `fcop-rules.mdc` frontmatter ↔ body changelog | `test_rules_metadata_consistency::test_fcop_rules_frontmatter_matches_body_changelog` |
| `fcop-protocol.mdc` frontmatter ↔ body changelog | `test_rules_metadata_consistency::test_fcop_protocol_frontmatter_matches_body_changelog` |

Future versioned metadata fields should pick up an equivalent
test in the same file.

### 3. Documentation freshness sweep

While auditing the codebase for the same *class* of inconsistency,
two pieces of public documentation were found stale at 0.7.0
(carried over silently to 0.7.1):

- `README.md` / `README.zh.md` "Recent releases" tables — last
  entry was 0.6.5 — extended to include 0.7.0, 0.7.1, 0.7.2.
- `mcp/README.md` "Stability" section and the "Already on `0.6.x`"
  upgrade pointer — extended to also mention `0.7.x` as a current
  release line.

Neither is a runtime bug; both made the landing page lie about
which protocol version a new visitor would land on. Fixed here so
0.7.2 ships with the public docs in sync with the published wheels.

## Why no PyPI yank, again

Same answer as `0.7.1.md`. Short version:

| Install command | Resolves to before yank | Resolves to after yank |
|---|---|---|
| `pip install fcop-mcp` (no pin) | **0.7.2** | 0.7.2 |
| `pip install -U fcop-mcp` | **0.7.2** | 0.7.2 |
| `uvx fcop-mcp` (no `--refresh`) | cached `0.7.0`/`0.7.1` if previously installed; otherwise **0.7.2** | same |
| `uvx --refresh fcop-mcp` | **0.7.2** | 0.7.2 |
| `pip install fcop-mcp==0.7.1` | 0.7.1 | 0.7.1 (yank does not block explicit pins, PEP 592) |

Yank does **not** repair anyone whose `uvx` cache is currently on
0.7.0 or 0.7.1; only a `uvx --refresh fcop-mcp` does. So yank
buys nothing for the actually-broken case (cached 0.7.0
`ImportError`), while removing legibility from the version history
(0.7.1 *was* a real attempted release; future debuggers reading the
PyPI page deserve to see why 0.7.2 immediately followed). Yank is
kept available as a reversible last resort.

## Verifying 0.7.2 locally

```bash
uvx --refresh fcop-mcp@0.7.2 --version
# → fcop-mcp 0.7.2

# Inside any FCoP-managed project, after the MCP server is registered:
#   fcop_report()
# should now print:
#   fcop-mcp:  0.7.2
#   fcop:      0.7.2
#   rules:     1.8.0   (both columns)
#   protocol:  1.6.0   (both columns)
```

If `rules: 1.7.0` still appears in either column, you are still on
the 0.7.1 wheel — `pip install -U` / `uv cache clean fcop fcop-mcp`
+ `uvx --refresh` to fix.

## What is intentionally **not** in 0.7.2

- **`audit_drift()` ADMIN/SYSTEM exemption** (REPORT-006 dogfood
  finding). Currently `audit_drift()` reports `ADMIN`-via-`ME`
  transcription as a `session_id ↔ role` conflict because it does
  not exempt the framework roles. Tracked separately; not in this
  patch because changing audit semantics is a behaviour change and
  0.7.2 is metadata-only by charter.
- **`docs/upgrade-fcop-mcp.md` / `docs/release-process.md`
  modernization**. Both still describe the 0.6.x release flow.
  Out of scope for 0.7.2; the new authoritative checklist is
  [`RELEASE-CHECKLIST.md`](./RELEASE-CHECKLIST.md). Will be
  consolidated in a later docs-only patch.
