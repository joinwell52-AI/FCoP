# fcop + fcop-mcp 1.2.1 Release Notes

**发布日期**：2026-05-11  
**PyPI**：[fcop 1.2.1](https://pypi.org/project/fcop/1.2.1/) · [fcop-mcp 1.2.1](https://pypi.org/project/fcop-mcp/1.2.1/)

---

## 为什么这个版本存在

v1.2.1 是一个**锁步对齐补丁**（lockstep restore patch）。

在 v1.1.1 / v1.2.0 阶段，`fcop-mcp` 经历了多次手动补发，绕过了 CI `release.yml` 中规定的两包同版校验，导致：

- `fcop` 停留在 `1.1.0`，`fcop-mcp` 已到 `1.2.0`
- 两包版本不一致，违反 ADR-0002 锁步规则

v1.2.1 将 `fcop` 从 `1.1.0` 空 patch 升至 `1.2.1`，与 `fcop-mcp 1.2.1` 严格对齐，**后续所有发版均须通过 CI 一条龙（tag push → release.yml → PyPI + GitHub Release）**。

## v1.2.0 功能回顾（随本次锁步一并稳定交付）

### 治理事件审计层（ADR-0030-bis Layer 1）

**新增 2 个 MCP 工具**（总数 30 → 32）：

#### `list_governance_events`

读取 `fcop_events.jsonl` append-only 治理日志，支持按 `risk`（Safe / Sensitive / Critical）和 `tag`（ALLOW / REVIEW_TAG / CRITICAL_TAG）过滤；每条记录包含 tool、risk、tag、args_hash、session_id、时间戳。

#### `get_governance_summary`

汇总统计：总调用量 / 各风险层分布 / Top 10 工具 / CRITICAL_TAG 事件清单；治理健康检查快速入口。

### 新增 `fcop_mcp.governance` 子包

| 组件 | 职责 |
|---|---|
| `FCoPGovernanceMiddleware` | FastMCP `on_call_tool` hook；每次工具调用自动执行 Skill Resolver → Risk Tag → Event Log → call_next（不阻断） |
| `skill_registry.yaml` | 全部 32 个内置工具的 risk 分类配置；支持项目级覆盖 |
| `resolve_skill` | 单独使用的 Skill 解析工具函数 |
| `emit_event` | 单独使用的事件写入工具函数 |

### 设计定位（ADR-0030-bis SMB 原则）

- **行为账本，不是防火墙**：记录是强制的，阻断是可选的（v2.x 再加）
- **stateless logging only**：无 approval_token、无 policy engine、无状态机
- **Authority Rule**：风险分类决策在 MCP Interceptor，不在 Agent 执行层（ADR-0030 §四）

### `fcop_check()` 治理审计整合

输出末段新增「治理事件日志」摘要，显示 CRITICAL_TAG 操作清单（无对应 Task + Review 即为治理缺口）。

### 测试

新增 23 个 governance 单元测试（`tests/test_fcop_mcp/test_governance.py`），覆盖 Skill Resolver、Event Logger（线程安全）、Middleware on_call_tool 合约。

## 发版流程修正说明

本版本同时修正了发版流程漏洞：

> **规则**（已写入 `docs/releases/RELEASE-CHECKLIST.md`）：  
> 所有 MINOR/PATCH 发版必须以 `v{version}` tag push 触发 `release.yml` CI，不允许手动 `twine upload` 绕过。  
> `fcop` 与 `fcop-mcp` 版本号必须在同一 tag 推送时保持一致（lockstep）。
