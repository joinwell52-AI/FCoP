## 运行时影响评估

### `fcop` 包（Python 库）

**无任何变更**。`fcop` 源码自 v1.3.0 tag 后 **零改动**，PyPI `fcop==1.3.0` 完全准确。

### `fcop-mcp` 包（MCP server）

涉及 `mcp/src/fcop_mcp/server.py` 的两个 commit：

#### `b4aeb8c` — 删除过期 `# type: ignore` 注释

```python
# 变更前（v1.3.0 tag）
report = project.audit(  # type: ignore[attr-defined]
    scope=scope,
    output=output,
)

# 变更后（当前 main）
report = project.audit(
    scope=cast(Literal[...], scope),
    output=cast(Literal[...], output),
)
```

**运行时影响**：Python 的 `cast()` 是纯类型标注函数，运行时等价于 identity（直接返回原值）。行为 **完全等价**，无任何运行时差异。

#### `8f11d20` — 同上（是 `b4aeb8c` 的后续完整版，两者合计即最终状态）

## ADMIN 决策选项

### 选项 A：**不发 1.3.1**（推荐）

**理由**：
- 两个包运行时行为与 1.3.0 完全相同
- `cast()` 是 type-checker 工具，`fcop-mcp/server.py` 是 entry-point 文件，终端用户不 import 它
- `uv.lock` 是开发工具文件，不随 wheel 打包
- 文档修正已在 main，用户可从源码读到

**操作**：无需任何操作。

---

### 选项 B：**发 1.3.1**

**理由**：
- 保持 main == PyPI 的洁净原则
- `mcp/uv.lock` 锁定 fcop 1.1.0 可能让从源码 `uv run` 的用户感到困惑

**操作步骤**（如 ADMIN 决定发版）：

```bash
# 1. 同步 version 到 1.3.1
# src/fcop/_version.py 保持 1.3.0（fcop 无变化，不动）
# mcp/src/fcop_mcp/_version.py: 1.3.0 → 1.3.1

# 2. 更新 CHANGELOG / docs/releases/1.3.1.md

# 3. 构建 + 校验 fcop-mcp wheel（fcop 不重发）
cd mcp && uv build
twine check dist/fcop_mcp-1.3.1*

# 4. tag + push（触发 CI release.yml）
git tag v1.3.1 && git push origin v1.3.1
```

> **注意**：选项 B 下 `fcop` 不需要重发（无代码变更）；仅 `fcop-mcp` 从 1.3.0 → 1.3.1。
> 这会短暂打破锁步（fcop=1.3.0, fcop-mcp=1.3.1），需在 CHANGELOG 注明。
