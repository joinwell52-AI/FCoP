# Release 0.6.6 — Documentation-only sync (no PyPI bump)

- **Date**: 2026-04-27
- **PyPI**: **no upload** — `fcop` and `fcop-mcp` stay at **0.6.5** on
  PyPI. `_version.py` is **not** bumped. The next code release (whatever
  number it takes) will carry the docs forward.
- **Why a release note for a doc-only patch?** Because `mcp/README.md`
  is the **PyPI long description** for `fcop-mcp` — every doc edit there
  affects how new users see the package the next time we upload. Pinning
  the cross-surface sync to a numbered note makes the audit trail
  searchable.
- **Stability**: no signatures changed, no behavior changed, no test
  changed. `tool_surface.json` and `public_surface.json` snapshots
  unchanged.

---

## What this patch fixes

A self-audit after the 0.6.5 release found three drift points where
0.6.4 / 0.6.5 changes had not propagated cleanly across the
`fcop` / `fcop-mcp` documentation surfaces. Symptom: someone reading
the package on PyPI or the GitHub repo would see a `0.6.5` wheel but
no human-readable explanation of why `new_workspace` started prepending
a Rule 0.a.1 reminder.

### 1. `docs/mcp-tools.md` — wrong resource count

- **Before**: "工具 26 个 / 资源 **10 个**" (last touched in 0.6.3).
- **After**: "资源 **12 个**" (9 static URIs + 3 templates) — the
  count now matches `tests/test_fcop_mcp/snapshots/tool_surface.json`
  and the `LETTER-TO-ADMIN` letter.
- **Resource table**: added two missing rows for the 0.6.4 resources
  `fcop://prompt/install` and `fcop://prompt/install/en`.
- **Per-tool notes**: appended one-liner 0.6.5 behaviour notes to the
  `fcop_report` and `new_workspace` rows so the index reflects the
  Rule 0.a.1 tripwires.

### 2. `mcp/README.md` (= PyPI `fcop-mcp` long description) — silent on 0.6.5

- **Before**: the "What can the server actually do?" lede mentioned
  the 0.6.4 changes (2 new resources + `force` parameter) but stopped
  there — readers landing on PyPI for `fcop-mcp 0.6.5` saw no mention
  of the Rule 0.a.1 tripwires.
- **After**: appended one bilingual sentence explaining the 0.6.5
  tool-layer wiring (`new_workspace` reminder + `fcop_report` four-step
  template), with a link to `docs/releases/0.6.5.md` for full detail.

### 3. `LETTER-TO-ADMIN.{zh,en}.md` — top summary block silent on 0.6.5

- **Before**: header block titled "0.6.4 摘要" / "0.6.4 in one block:".
- **After**: kept the heading verbatim (so `tests/test_fcop/test_rules.py::test_zh_contains_064_summary_marker`
  still passes), added one bullet at the end of the block describing
  the 0.6.5 Rule 0.a.1 tripwires in plain language ("agent 跳步会被即时
  提醒，不阻塞").
- **Slice safety**: both edits land **before** the second `---`
  horizontal rule, so `get_letter_intro()` continues to surface the
  new bullet to ADMIN at init time.

---

### 4. Root `README.md` / `README.zh.md` — "0.6.3 highlights" long paragraph rewritten

- **Before**: a single ~10-line paragraph mixing five different
  concerns (install pointer, upgrade pointer, tool-index pointer,
  0.6.3 highlights, "wrong distribution" warning), with version-
  specific text frozen at 0.6.3. Adding 0.6.4 / 0.6.5 inside it
  would have made it unreadable.
- **After**: split into three discrete blocks, none baked to a
  specific version.
  - **Pointers table** (5 rows): "you want to do X → go to Y", maps
    install / agent-install / upgrade / tool index / changelog.
  - **Recent releases table** (3 rows): one line per version (0.6.3 /
    0.6.4 / 0.6.5), each linking to its `docs/releases/<v>.md`. This
    table is the only version-coupled element and is meant to be
    truncated / extended in place each release.
  - **"Wrong fcop on PyPI" warning**: pulled out as a standalone
    blockquote so it stops competing with the install pointer.
- **Why it counts as docs-only**: same 5-pointer set as before, no
  new package promises, no new external commitments. The release
  table is purely a reordering of content already in
  `docs/releases/`.

## What this patch deliberately does NOT do

- **Does not bump `_version.py`.** Both packages stay at 0.6.5 on PyPI.
- **Does not change any tool / resource signatures.** The drift was
  purely in human-readable copy.
- **Does not touch the existing `docs/releases/0.6.3.md`,
  `0.6.4.md`, `0.6.5.md` files.** Those are historical records and
  remain untouched by this patch.

## Rationale

This patch is the corrective action item from the user-driven
self-audit logged in chat on 2026-04-27, which asked: **"自我检查所有
文档、GitHub、PyPI，看下是否逻辑一致；fcop-mcp 那么多功能，究竟什么
作用？"** ("Self-audit all docs / GitHub / PyPI for logical consistency;
what is FCoP-MCP's many functionalities actually for?")

The audit found one numerically wrong claim (`docs/mcp-tools.md` said
"10 resources" while the truth was 12) and two zero-mention surfaces
(PyPI long description and the customer-facing Letter both silent on
0.6.5). All three are now patched. The "what FCoP-MCP actually does"
question is answered structurally elsewhere — see
[`docs/mcp-tools.md`](./../mcp-tools.md) for the seven-category
breakdown of the 26 tools. Two of those tools (`fcop_report`,
`set_project_dir`) are the only ones ADMIN ever names directly; the
other 24 the agent picks itself.
