# Release 0.7.1 — Pin hotfix + sub-agent / tripwire / Rule-5 cleanup

**Date**: 2026-04-27 (same calendar day as 0.7.0; this is a hotfix)
**Packages**: `fcop` 0.7.0 → 0.7.1, `fcop-mcp` 0.7.0 → 0.7.1
**Protocol bumps**: `fcop_protocol_version` 1.5.0 → 1.6.0,
`fcop_rules_version` 1.7.0 → 1.8.0
**Surface compatibility**: ADR-0003 minor-boundary rules apply —
purely additive (new dataclasses, one new MCP tool); no deprecations,
no removals.
**Yanked**: `fcop-mcp 0.7.0` is being pulled from PyPI; users on
0.7.0 must upgrade.

## Why this release exists

Three threads converged on the afternoon of 2026-04-27, and ADMIN's
verdict was unambiguous: *"什么乱七八糟的，0.7.1一起搞完！！！！
不要残留，最后又不对！！！"* — finish them all in one cycle, don't
let any residue leak into 0.7.2.

### Thread A — the pin bug (`ISSUE-20260427-006`, release blocker)

`fcop-mcp 0.7.0` was published with `mcp/pyproject.toml` carrying
the stale dependency pin `"fcop>=0.6,<0.7"`. The bump in
`mcp/src/fcop_mcp/_version.py` was correct, the import statements
in `server.py` were correct, but the dependency pin was a leftover
from the 0.6.x line. `uvx fcop-mcp` therefore resolved
`fcop-mcp 0.7.0` together with `fcop 0.6.5`, and the very first
`fcop_report()` call crashed with `ImportError: cannot import name
'RoleOccupancy' from 'fcop'` — `fcop-mcp 0.7.0` imports
`RoleOccupancy`, `Project.role_occupancy()`, and `OccupancyState`
from APIs that only exist in `fcop 0.7.0`. Workaround was a manual
`uv cache clean` plus pinning `fcop==0.7.0` by hand; that is not a
shippable user experience.

The fix is mechanical (correct the pin), but the deeper question is:
how did this slip past us when the version bump itself was
intentional and reviewed? Two reasons:

1. **No automated coupling check.** Nothing in CI verified that
   `fcop-mcp X.Y.Z`'s `fcop` dependency lower-bound matches
   `X.Y`. The only "check" was human eyeballs on a `pyproject.toml`
   diff that also touched a description string and a version literal
   on the same lines.
2. **No fresh-install sanity gate.** The release pipeline did not run
   `uvx --refresh fcop-mcp@<just-uploaded>` from a clean cache after
   upload. Local developer environments had `fcop 0.7.0` already
   installed in editable mode, so the import worked locally and
   masked the resolver bug.

Both gaps now have explicit countermeasures in this release; see
[Release infrastructure](#release-infrastructure) below.

### Thread B — sub-agent role impersonation (`ISSUE-20260427-004`)

While verifying 0.7.0, ADMIN dropped a copy of
`AMEND-20260427-011-REPORT-20260427-010-011-FLOW.md` from another
project and asked: *"the agent is also upgrading itself, so it has
its own sub-agents — multi-process. But task-wise, it should
finish its own task; even if it spawns several sub-agents, it must
not stand in for other roles."*

The on-disk evidence in that ledger showed exactly this pattern: a
single agent had spawned sub-processes that wrote a `REPORT-*` file
under a *different role code* than the parent session's binding.
Rule 1's role-uniqueness clause from 0.7.0 was sound — but it was
silent on what happens *inside* one agent's address space. A
parent session bound to role A could (and did) tell its sub-agent
"now write a report as role B", and the protocol had no
counter-argument.

The 0.7.1 fix is two-layered:

1. **Protocol clarification (`fcop-rules.mdc` Rule 1).** Sub-agents
   and worker sub-processes inherit the parent session's role.
   "Even temporarily" is not a legal escape hatch.
2. **Detection (`Project.audit_drift()` + `fcop_check()`).** Walk
   every `TASK-*` / `REPORT-*` / `ISSUE-*`, group frontmatter
   `session_id` by role; any session that signs files under more
   than one role is direct evidence of impersonation. The audit is
   surfaced both as a standalone tool (`fcop_check`) and inlined
   into the UNBOUND branch of `fcop_report()` so ADMIN sees
   pre-existing damage before assigning new roles.
3. **Soft MCP-level lock.** `write_task` / `write_report` /
   `write_issue` remember the first `sender` they see in a single
   MCP process and warn (plus drop a `.fcop/proposals/role-switch-*.md`
   evidence file) when a later call switches role. The lock is
   intentionally **soft** — agents that *need* to bypass it for
   legitimate handoffs can, but the bypass leaves a trail.

We discussed hard-blocking and decided against it. A hard block can
be hidden by a sufficiently motivated agent (just don't use the
tool); a loud warning that drops on-disk evidence cannot be hidden
without obvious tampering. FCoP's design philosophy is *the file is
the protocol* — make tampering expensive, not impossible.

### Thread C — `AMEND-*` / `-v2` was never implemented (`ISSUE-20260427-005`)

The same `AMEND-20260427-011-...` file surfaced a smaller but
embarrassing problem: Rule 5 of `fcop-rules.mdc` had been listing
`AMEND-NNN-correcting-NNN.md` and `REPORT-NNN-v2.md` as canonical
correction patterns since 1.x, **but the `fcop` library has never
shipped a parser for either**. Filenames in those formats land in
the ledger but are invisible to `Project.list_*`, `role_occupancy()`,
and the new `audit_drift()`. Documentation said "do this", code
said "no idea what that file is".

The 0.7.1 fix is to delete the lie. Rule 5 now states corrections
are made by allocating the next `REPORT-NNN` / `TASK-NNN` sequence
number and cross-referencing the corrected file in frontmatter
(`amends:` / `supersedes:`). Sequential filenames have always
worked; we just stop pretending the alternative paths exist.

### Thread D — Rule 0.a.1 tripwire was always blind (`ISSUE-20260427-001`)

A previously deferred issue: the `new_workspace` tripwire in
`fcop-rules.mdc` Rule 0.a.1 only catches `init_project`-time drift.
Once a project is initialized, manual `git commit -am`, raw
`echo > foo.md`, IDE-side edits, and external scripts can land
freely without going through `task → do → report → archive`. The
0.7.1 fix is `Project.audit_drift()` itself: the working-tree
audit reads `git status --porcelain -z`, removes everything under
`docs/agents/{tasks,reports,issues,log}`, and reports the rest as
"work that landed without a TASK". Rule 0.a.1's text now also
explicitly states it binds *all* write paths, not just MCP tools —
shell, git, IDE, scripts.

## What changed

### Library (`fcop`)

#### Added

- `Project.audit_drift() -> DriftReport`. Two-stream audit
  (working-tree drift + session/role conflicts).
- `fcop.models.DriftEntry`, `fcop.models.SessionRoleConflict`,
  `fcop.models.DriftReport` — frozen dataclasses, exported from
  the top-level `fcop` package.

### MCP server (`fcop-mcp`)

#### Fixed

- `mcp/pyproject.toml` `dependencies` corrected from
  `"fcop>=0.6,<0.7"` to `"fcop>=0.7,<0.8"`
  (`ISSUE-20260427-006`).

#### Added

- `fcop_check(lang="zh")` MCP tool — wraps `audit_drift()` and
  renders the report in either Chinese or English.
- Per-MCP-process role lock in `write_task` / `write_report` /
  `write_issue`. First `sender` wins; subsequent role switches
  surface a Rule-1 warning and drop a `.fcop/proposals/role-switch-*.md`
  evidence file.
- `fcop_report()` UNBOUND branch now folds in a one-screen audit
  summary.

### Protocol

- `fcop_protocol_version`: 1.5.0 → 1.6.0
- `fcop_rules_version`: 1.7.0 → 1.8.0
- Rule 1 — sub-agent identity inheritance clause added.
- Rule 0.a.1 — applies-to-all-write-paths clarified.
- Rule 5 — `AMEND-*` / `-v2` examples removed; sequential
  corrections only.
- `LETTER-TO-ADMIN.{zh,en}` — "a sub-agent is not an extra role"
  warning added next to "one role, one agent".

### Release infrastructure

- `tests/test_fcop/test_pyproject_pins.py` (new). Reads
  `mcp/pyproject.toml` at test time, parses the `fcop` dependency
  spec, and asserts the lower bound matches `fcop-mcp`'s minor.
  Failing this test blocks any future release where the pin drifts.
- `adr/ADR-0002-package-split-and-migration.md` — new "Lockstep
  pin rule" section formalises the constraint that
  `tests/test_fcop/test_pyproject_pins.py` enforces, plus a
  "发版前 lockstep 检查表" (pre-release lockstep checklist).
- `docs/releases/RELEASE-CHECKLIST.md` (new). Eight-phase release
  walkthrough. Phase 6 includes a mandatory `uvx --refresh
  fcop-mcp@X.Y.Z` from a clean cache before declaring the release
  done — the gap that masked `ISSUE-20260427-006`.

## Post-mortem — `ISSUE-20260427-006`

**Timeline (UTC+8, 2026-04-27)**

| Time  | Event |
| ----- | ----- |
| 14:xx | `fcop 0.7.0` and `fcop-mcp 0.7.0` uploaded to PyPI (lockstep). |
| 15:xx | First user `uvx fcop-mcp` from a fresh cache; `fcop_report()` raises `ImportError: cannot import name 'RoleOccupancy' from 'fcop'`. |
| 15:xx | Workaround found: `uv cache clean fcop-mcp --force` → "no cache entries found" (uv was looking at the wrong cache key); only `pip install fcop==0.7.0 fcop-mcp==0.7.0` recovered. |
| 16:xx | Root cause identified: `mcp/pyproject.toml` still pins `fcop>=0.6,<0.7`. Resolver picks `fcop 0.6.5` for fresh installs. |
| 16:xx | ADMIN: *"什么乱七八糟的，0.7.1一起搞完！！！！"* — scope expands to roll up `ISSUE-001 / -004 / -005` in one release. |
| 17:xx | 0.7.1 cycle complete (this document). |

**Root cause**: human pin update missed during the 0.6.5 → 0.7.0
bump, with no automation to catch it.

**Contributing factors**

1. The `pyproject.toml` diff for the bump was multi-line (description
   string mentioning the version, version literal, and the
   dependency pin all changed together) — it's the kind of diff
   where one of three intended edits silently doesn't happen.
2. Local dev environments had `fcop` installed in editable mode, so
   the resolver bug never fired during local testing.
3. No "fresh-cache install" gate after publish.

**Prevention (now in place)**

1. `tests/test_fcop/test_pyproject_pins.py` reads the pin at test
   time and fails the build if it doesn't match `fcop-mcp`'s minor.
2. `docs/releases/RELEASE-CHECKLIST.md` Phase 6 mandates a
   post-publish `uvx --refresh fcop-mcp@X.Y.Z` smoke test from a
   clean cache.
3. `adr/ADR-0002` "Lockstep pin rule" makes the constraint explicit
   in protocol docs, so future maintainers don't have to
   reverse-engineer it from CI failures.

**What we are not doing (and why)**

- **Hard-blocking sub-agent role impersonation.** Soft warnings +
  on-disk evidence files were chosen instead. Rationale: a hard
  block can be silently bypassed (don't call the tool); a loud
  warning that lands as a tracked file on disk cannot. *The file
  is the protocol.*
- **Auto-yank on ImportError.** PyPI yanks are reversible but
  noisy; we prefer manual yanks at release time when the
  maintainer can confirm scope.

## Upgrade notes

- `pip install -U fcop fcop-mcp` (or `uvx --refresh fcop-mcp`).
- If you were on `fcop-mcp 0.7.0` and hit the `ImportError`, the
  upgrade is the fix — no workaround needed.
- If you have local AGENTS.md / CLAUDE.md / .cursor/rules/ deployed
  from a previous version, run `redeploy_rules` (MCP tool) once
  after the upgrade to pick up `fcop_rules_version: 1.8.0` and
  `fcop_protocol_version: 1.6.0`.
