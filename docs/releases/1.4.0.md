# fcop-mcp 1.4.0 Release Notes

> **Breaking-ish behavior change** — write-side tools now **require** an
> explicit project binding; they will refuse with `WriteRefused` if only a
> cwd fallback is available. See the upgrade guide below.

---

## 背景 / Background

### PM #50 事件（2026-05-12）

PM 角色的 MCP Server 的 cwd 被设定为 `C:\Users\Administrator`（USER HOME）。
在未调用 `set_project_dir()` 的情况下，PM 直接调用了 `redeploy_rules()`。
`fcop-mcp 1.3.x` 的解析瀑布最终用 `cwd` 兜底，将 `fcop-rules.mdc` 等文件写入到
了 USER HOME 目录（`C:\Users\Administrator\.cursor\rules\`），造成系统级污染。

对应 TASK: `fcop/tasks/TASK-20260512-001-ADMIN-to-ME.md`

---

## 本版本变更 / Changes in 1.4.0

### D1 — Write-side 工具显式绑定守门

**影响范围：所有 write-side MCP 工具（15 个）**

```
init_project      init_solo         create_custom_team
write_task        archive_task      write_report
write_issue       write_review      mark_human_approved
deploy_role_templates               new_workspace
drop_suggestion   fcop_audit        redeploy_rules
fcop_create_alert
```

**新行为：**

1. 若工具调用时**没有任何显式绑定**（即 `set_project_dir()` 未调用，
   且 `FCOP_PROJECT_DIR` / `CODEFLOW_PROJECT_DIR` 均未设定），
   工具直接抛出 `WriteRefused` 错误，不再允许 cwd 兜底。

2. 自动标记探测（`auto:{marker}` 源）仍然被允许。只有完全退化到
   `"cwd fallback"` 的情况才被拦截。

**错误信息格式：**
```
WriteRefused: <tool_name> requires an explicitly bound project root,
but no binding is active (resolved via 'cwd fallback …').
Call set_project_dir(path=...) first, or restart the MCP server
with cwd set to the project root.
```

---

### D2 — Protected Path 拒绝列表

即使通过 `set_project_dir()` 显式绑定，以下路径**仍会被拒绝**：

| 类别 | 示例（Windows） | 示例（Unix） |
|---|---|---|
| USER HOME 及父目录 | `C:\Users\<name>`, `C:\Users` | `~`, `/home` |
| APPDATA / LOCALAPPDATA | `%APPDATA%`, `%LOCALAPPDATA%` | — |
| PROGRAMDATA | `%PROGRAMDATA%` | — |
| 驱动器根目录 | `C:\`, `D:\` | `/` |
| Unix 系统目录 | — | `/etc`, `/usr`, `/opt`, `/bin` … |

**错误信息格式：**
```
WriteRefused: <tool_name>: project_dir resolved to a protected system path
(<path>). FCoP write-side tools refuse to operate at this scope to prevent
global pollution. Bind to a project subdirectory instead via
set_project_dir(path=...).
```

---

### D3 — `fcop_report()` / `unbound_report()` 顶部告警

当 `fcop_report()` 检测到以下情况时，会在输出顶部插入 ⚠️ 告警：

- `project_path` 解析到受保护的系统路径
- `source` 为 `cwd fallback`（无显式绑定）

告警内容提示 ADMIN 调用 `set_project_dir()` 完成绑定。

---

### D4 — MCP Schema `binding_required` 标签

所有 write-side 工具在 `tools/list` 响应中携带 `tags: ["binding_required"]`，
供 Orchestrator 层自动检测哪些工具需要显式绑定。

---

### D5 — 测试覆盖

新增测试文件：`mcp/tests/test_write_guard.py`，覆盖 8 个测试场景：

| TC | 场景 |
|---|---|
| TC-01 | 无绑定 + cwd 兜底 → WriteRefused |
| TC-01b | cwd = HOME → WriteRefused |
| TC-02 | 显式绑定到 USER HOME → WriteRefused（D2）|
| TC-03 | 显式绑定到盘符根目录 → WriteRefused（D2，Windows 专项）|
| TC-04 | 合法项目目录 → 正常通过 |

## 升级指南 / Upgrade Guide

### 场景 A：通过 Cursor MCP Config 启动

在 MCP 配置中添加 `FCOP_PROJECT_DIR` 环境变量：

```json
{
  "fcop": {
    "command": "uvx",
    "args": ["fcop-mcp"],
    "env": {
      "FCOP_PROJECT_DIR": "/path/to/your/project"
    }
  }
}
```

### 场景 B：在会话开始时调用

```python
set_project_dir(path="/path/to/your/project")
```

ADMIN 在每次新 MCP 会话后（通过 `fcop_report()` 确认 source 为
`session:set_project_dir`），再发布正式 task。

### 场景 C：让 cwd 自动探测（仍受支持）

若 MCP Server 的 cwd 就是项目根目录，且目录中包含以下任一标记文件：

- `docs/agents/fcop.json`
- `.cursor/rules/fcop-rules.mdc`
- `docs/agents/tasks/`（目录存在）

则解析源为 `auto:{marker}`，**不属于 cwd fallback**，write-side 工具
**正常通过 D1 检查**。此时无需任何额外配置。

## 协议升级 / Protocol Changes

> `fcop_protocol_version` 从 **2.1.0** 升至 **2.2.0**（`fcop` 包）

### `supersedes:` Frontmatter 字段（TASK-004）

所有 IPC envelope（TASK / REPORT / ISSUE / REVIEW）新增可选字段：

```yaml
supersedes: TASK-20260418-010   # 单值
# 或
supersedes:
  - TASK-20260418-010
  - TASK-20260418-011            # 多值
```

**语义**：本文件作为对指定历史文件的"文件级修正"，旧文件保留磁盘（Rule 5 append-only），
`list_tasks` / `list_reports` 自动双向标注 `[supersedes X]` / `[superseded by X]`。

**与其他字段区分**：
- `supersedes:` — 文件级修正（Rule 5 兜底）
- `parent:` — 工作派生（子任务 / 子报告）
- `related:` — 交叉引用（无层级关系）

**Schema 同步**：`spec/schemas/ipc-envelope.schema.json` 已加入 `supersedes` 字段定义。

---

### GATE Design Pitfalls 章节（TASK-003）

`fcop-protocol.mdc` 新增 **`## GATE Design Pitfalls`** 节，归档 Bridgeflow 项目中发现的典型
GATE 设计陷阱，供未来项目参考：

**Pitfall 1 — GATE 描述自我命中（Self-Collision）**：

当 GATE 条件在语义上也能匹配"描述该 GATE 的 OPS 任务自身"时，GATE 会在审计时触发误报，
导致"陷阱无限循环"。根因：OPS 任务文件本身包含 GATE 关键词，被 GATE 的内容扫描误判为
被监控对象。修复方向：区分元数据层（OPS 文件本身）与内容层（被 GATE 监控的对象）；
推荐语义化实证条件，避免纯关键词扫描。

**后续**：`fcop_audit` D8 scan（`_scan_gate_self_collision()`）预留锚点，归 **v1.5** 实现。
