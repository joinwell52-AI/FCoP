## 实例场景

2026-04-27 ADMIN 测试外部项目时观察到：单一 session 被指派为 `PLANNER`
角色，PLANNER 派生子进程跑测试，**子进程独立落了一份报告**，frontmatter
的 `sender: CODE_EXPERT`——但 ADMIN 从未指派 `CODE_EXPERT`。

事后 PLANNER 发现自己代角色越权，自补一份 `AMEND-20260427-011-REPORT-...`
解释来龙去脉。这份 AMEND 文件本身又触发了 ISSUE-005（AMEND-* 不在协议解析
器覆盖范围内）。

## 根因

Rule 1（"两阶段启动"）只约束**呼叫者**的初始 BOUND——它没显式声明
"派生出去的子 agent 也继承同一个角色"。新一代 agent 普遍带 sub-agent
能力（Cursor / Claude Code / 各种 framework 的 worker），子 agent 不知道
自己应该沿用父会话的角色，自由心证地选了一个"贴合任务内容"的角色码，
写了文件就走。

`fcop` / `fcop-mcp` 的 `write_task` / `write_report` / `write_issue` 在
0.7.0 之前不做任何 `sender ↔ session_id` 的一致性校验——光看每份文件
单独看都"合法"，但拼起来=身份伪造。

## 已采纳的方向（0.7.1）

**双轨并举：协议 + 工具**。

1. **协议侧（Rule 1 / fcop-rules.mdc 1.8.0）**
   增加"子 agent 继承呼叫者身份"明文条款：派生子单元仍是同一席位，
   `sender` / `reporter` 必须沿用父会话被 ADMIN 指派的角色，禁止
   自命另一角色码"代写"。要让别的角色出报告只有 Rule 4 一条路：
   派 `TASK-*.md` 给那个角色，等 ADMIN 单独指派后再写。

2. **工具侧（fcop-mcp 0.7.1 / fcop_check）**
   新增 `fcop_check()` MCP 工具，做 **`session_id ↔ role` 一致性审计**：
   扫 `docs/agents/{tasks,reports,issues,log}/` 全部文件，按
   `session_id` 聚合，同一 `session_id` 出现在两个不同角色名下=
   身份冒充证据，输出直接列给 ADMIN。

3. **同步增强 `fcop_report()`**：在 Role occupancy 段下方增加
   "session_id cross-role conflicts" 子段，让 UNBOUND 阶段的 agent
   一开会就看到既有冲突。

`fcop_check()` 是**事后审计**而非事前阻塞——不阻塞 write_*，因为：
(a) 子 agent 通常不接 MCP，阻塞写不到子 agent 上；(b) 阻塞=隐藏证据，
事后审计=显化证据，后者更符合"文件即协议"。
