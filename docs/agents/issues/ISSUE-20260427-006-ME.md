## 实例场景

`fcop-mcp 0.7.0` 已发布到 PyPI 并已打 tag `v0.7.0`。但发布后用户
（ADMIN）反馈 `fcop_report()` 调用挂掉，提示找不到 `RoleOccupancy`
等新 API。

复现步骤：

1. `uvx --refresh fcop-mcp` 拿到最新 0.7.0。
2. uvx 解析依赖：`mcp/pyproject.toml` 写的是 `fcop>=0.6,<0.7`，于是
   uvx 给 `fcop-mcp 0.7.0` 配了 `fcop 0.6.5`。
3. `fcop-mcp 0.7.0` 的 `server.py` 已 import `fcop.models.RoleOccupancy`、
   调 `Project.role_occupancy()`——这两个 API 0.6.5 里**不存在**。
4. 启动报 `ImportError` / `AttributeError`，`fcop_report()` 完全不可用。

## 根因（双层）

**直接原因**：发版前 `fcop` 库版本号已从 0.6.5 → 0.7.0、并加了破坏性的
新 API（per ADR-0003 minor 边界允许）；但 `mcp/pyproject.toml` 里
`fcop>=0.6,<0.7` 这条依赖约束**没有同步上调**。`fcop` 包自己 0.7.0 没
事，`fcop-mcp` 0.7.0 单独看起来也"没问题"，但两者在 PyPI 上对接的
依赖图直接坏了。

**深层原因**：发版流程没有"`fcop-mcp` 的 minor 必须等于其 fcop 依赖
的 minor"这条 lockstep 强制检查。ADR-0002 写了这条规矩，但只是叙述，
没落到 CI 测试里。`mcp/pyproject.toml` 完全靠人记得改。

## 已采纳的方向（0.7.1）

1. **立即修复 pin**：`mcp/pyproject.toml` 把 `fcop>=0.6,<0.7` 改为
   `fcop>=0.7,<0.8`，`fcop-mcp` 版本号 0.7.0 → 0.7.1，重新打 wheel
   并发 PyPI。
2. **yank 0.7.0**：`twine` / `uv publish` 把 PyPI 上的 `fcop-mcp 0.7.0`
   yank 掉，避免新装用户继续踩坑。
3. **写自动化测试**：新增 `tests/test_pyproject_pins.py`，发版前断言
   `mcp/pyproject.toml` 里 fcop 的 lower bound 与当前包的 minor 匹配
   （`fcop-mcp 0.7.x` ⇔ `fcop>=0.7,<0.8`）。
4. **更新 ADR-0002**：示例段从写死的 `>=0.6,<0.7` 改为占位 +
   "lockstep 检查表"，强调 minor 同步是硬约束。
5. **新增 `docs/releases/RELEASE-CHECKLIST.md`**：发版一条龙必须勾的
   清单，第一条就是 pin 检查。

## 教训

`fcop-mcp` 与 `fcop` 在仓库里是 monorepo 同源，但在 PyPI 上是**两个
独立分发**，依赖图全靠 `pyproject.toml` 里那一行 pin 维系。这一行错
就直接 release 失败。CI 必须自动核对，不能信"人会记得"。
