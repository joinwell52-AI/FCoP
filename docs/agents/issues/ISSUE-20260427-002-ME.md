## 问题陈述

在 solo 模式（甚至任意模式）下，如果 ADMIN 在两个独立会话中都说「你是 ME」，协议没有任何机制拦住第二个 agent 取得 BOUND 状态。

## 最小复现

1. ADMIN 初始化 D:\FCoP 为 solo + ME（如本仓本 session 已做，commit `c1d272a`）。
2. ADMIN 开第一个 Cursor 窗口，agent A 调 `fcop_report` 后被告知 `你是 ME`，A 进入 BOUND，开始写任务。
3. ADMIN 另开第二个 Cursor 窗口，agent B 调 `fcop_report` 仅次看到「1 个角色 ME 可选」，ADMIN 又说 `你是 ME`，B 也进入 BOUND。
4. 现场：A、B 同时以 ME 身份写 `tasks/`、`reports/`、`issues/`。

## 5 个冷却装崩点（实验验证）

1. **文件名碰撞**：两人到同一时间都 `list_tasks()` 看到最大序号是 N，各自决定写 `TASK-{date}-{N+1}-ADMIN-to-ME.md`。后调者要么 `FileExistsError`、要么静默覆盖。
2. **路由键完全相同**：两份任务 `to-ME` 完全无法区分。`list_tasks(recipient="ME")` 同时把「应该 A 做的」与「应该 B 做的」返回给两人，各自以为都是自己的。
3. **Rule 0.b 自审失效**：Solo 模式下 0.b 靠「同一 agent 先写后读」两个视角解决伦理问题。两个 agent 都是 ME，A 写的提案 B 读到了——B 的「审查」对 A 的「提案」是**他审**、不是 0.b 要求的**自审**。Rule 0.b 在 solo 下的「用文件劈视角」那一招假设了单 agent。
4. **Rule 4 #6 事后触发**：两个 ME 一旦动同一 thread_key（哪怕未显式写，工具默认按文件名归同线程也行），立刻命中 Rule 4 #6 「two parallel drivers」违规——但**这是事后判定**，没有人在两边同时 `write_task` 时拦截。
5. **`fcop.json` 不会变**：`fcop.json` 是身份快照，不是会话状态。它说「ME 存在」，不说「ME 现在在不在线、被几个 session 拿着」。

## 什么是「指南针」

ADMIN 本人明确的纪律语言（2026-04-27 会话）：

> 「这里问题是，如果有两个一样的 ME，那么犯错的是 admin；我们要从落盘的文件去分析，因为文件即协议。」
>
> 「每个 agent，都要 admin 确认身份，但是 admin 不能把同一身份给多 agent！」

三条推论：

1. 「角色被谁占着」≡「磁盘上是谁在用这个角色码写文件」。`tasks/`、`reports/`、`issues/`、`log/` 是唯一权威源。
2. ADMIN 双 bind 是 ADMIN 的失误，但协议必须接住。agent 转 BOUND 前自己翻文件账本，冲突时拒绝。
3. 「临时 / temp」不是 bypass 通行证。任何要写盘的 agent 要拿不重名的角色码。

合并后的 Rule 1 不变量：

> **角色 ↔ 身份 是一对一映射。ADMIN 是不可借出的人类角色（→ 0 个 agent）；其他每个角色至多 1 个 agent；身份必须由 ADMIN 显式指派且不得重复发放。**

## 为什么「从磁盘推」是唯一合适的防线

FCoP 架构公理（`fcop-protocol.mdc` Architectural Principle）：

> The protocol lives on the filesystem. MCP tools are an ergonomic layer that expands into file operations. They must never become the source of truth.

上一轮我草拟过一个 0.6.7 候选 B：`.fcop/sessions/<role>.lock` 文件型 lease。**ADMIN 明确否定**。理由同上：那是把「角色占用」塞进**运行时**。FCoP 的全部存在意义恰恰是不要这样做。

代之：`fcop_report` UNBOUND 输出中添加「角色占用」段，逐角色统计 open tasks/reports/issues + archived + last session_id seen。agent 以这段为依据，事前自检；冲突时按 Rule 8 拒绝，落 `.fcop/proposals/`。
