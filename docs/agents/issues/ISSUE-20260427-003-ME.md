## 问题陈述

`fcop-mcp` 的 `write_task` / `write_report`（很可能也包含 `write_issue`）在计算下一个 `seq` 时，只扫各自的 `docs/agents/{tasks,reports,issues}/`，不扫 `docs/agents/log/{tasks,reports,issues}/`。当某类文件被 `archive_task` 移到 `log/` 后，下一次写入从 **001 重新起**，产生与已归档文件同基名的冲突。

**实证**：本会话连续两次踩同型 bug——

- `write_task` 调用：返回 `TASK-20260427-001-ADMIN-to-ME.md`，但 `log/tasks/` 里已有同名（已被人工 rename 抢救为 002）。
- `write_report(task_id="TASK-20260427-002")` 调用：返回 `REPORT-20260427-001-ME-to-ADMIN.md`，但 `log/reports/` 里也有同名（被人工 rename 抢救为 002 以与 TASK-002 配对）。

`write_report` 的 `task_id` 参数被正确写进了 frontmatter `references` 字段，但**文件名的 seq 是独立计数器，仍从 `reports/` 起，不看 `log/reports/`**。这进一步说明：**任务、报告、issue 的 seq 计数器各自独立，但都犯同样的"忽略 log/"错误**。

## 实际现场

2026-04-27 本会话：

- 14:25 左右：`write_task` → `TASK-20260427-001-ADMIN-to-ME.md`（0.6.6 docs patch 追溯记录）
- 14:30：`archive_task("TASK-20260427-001")` → 文件移入 `log/tasks/`，连带 `REPORT-20260427-001-ME-to-ADMIN.md` 一并归档
- 14:38：`write_task`（本任务 Rule 1 硬化）→ 返回 `TASK-20260427-001-ADMIN-to-ME.md` **（同名 #1）**
- 14:42：`write_report(task_id="TASK-20260427-002")`（本任务的报告）→ 返回 `REPORT-20260427-001-ME-to-ADMIN.md` **（同名 #2）**

同一仓同一天出现两份同基名任务、两份同基名报告。难调、难追溯、追溯工具（`grep -r TASK-20260427-001`）会同时命中两个不同质的实例。

本轮人工修正：

- `Move-Item tasks/TASK-20260427-001-ADMIN-to-ME.md tasks/TASK-20260427-002-ADMIN-to-ME.md`
- `Move-Item reports/REPORT-20260427-001-ME-to-ADMIN.md reports/REPORT-20260427-002-ME-to-ADMIN.md`（与 TASK-002 配对）

但这是现场抢救，不是修复。

## 为什么重要（不仅仅是其他 issue 里「双 ME 崩点 #1」的同型）

这与 ISSUE-002（双 ME）是同一架构缺陷的两面：

- ISSUE-002：「角色占用状态必须从磁盘账本推得」——但 `tasks/`/`reports/`/`issues/` 在归档后只剩半部账本，可观测性不完整。
- ISSUE-003（本 issue）：同一半部账本被序号生成器忽略，导致归档后再写产生同名碰撞。

两个 issue 都指向同一件事：**`<type>/` 不是账本的全部；`log/<type>/` 也是**。任何按「当前状态」做决定的逻辑都必须 union 现役与已归档。

## 候选修复方向

- **A【推荐】seq scan 同时包括 `log/<type>/`**：所有写盘工具（`write_task` / `write_report` / `write_issue`）计算 next_seq 时 union active 与 archived 两路径下同日期、同发送/报告方、同接收方的所有文件。改动量小、语义保守、向后兼容。
- **B 补充**：`archive_task` 在移走文件后在原目录里留一份 `.archived/<id>.tombstone` 零字节占位，list/seq 扫描认他。劣势：引入隐藏文件，不推荐。
- **C 零代码**：在 `letter-to-admin.md` 中加注「归档后勿在同一天同一发收方 `write_*`」。这是把 bug 倒同意书化，不推荐。

推荐 A。三个写盘工具应共享同一段 `_next_seq()` 辅助函数，估计代码量 < 30 行（含测试）。可作为 0.7.0 独立小修复，不与 Rule 1 硬化耦合。

## 验收标准

- 在 `tasks/` 为空且 `log/tasks/` 中存在 `TASK-{date}-001-ADMIN-to-ME.md` 时，`write_task(date=date, sender=ADMIN, recipient=ME, ...)` 必须返回 `TASK-{date}-002-ADMIN-to-ME.md`。
- 同型断言对 `write_report` 与 `write_issue` 同样成立。
- 现有 587 passed 基线不退。
