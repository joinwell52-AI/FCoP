## 实例场景

2026-04-27 本 session：ME 在未走 `new_workspace`、也未走 `fcop_report` 的情况下，完成了两个 commit（`274ce06` + `87e178a`）的跨表面文档修改。违反 Rule 0.a.1（全程未先写 TASK、未后写 REPORT、未归档），但 0.6.5 部署的两道 tripwire 都未触发。

## 根因

0.6.5 的 Rule 0.a.1 提醒被烘烤进两个点：

- `new_workspace`：开新工作区时 prepend 提醒。
- `fcop_report`（initialised 分支）：报告末尾附四步模板。

但「干活」不一定走这两道：

- 手工编辑文件（`StrReplace` / `Write` 类工具） → 不调用任何 fcop-mcp 工具。
- 外部 `git commit` / `git push` → 不经 fcop-mcp。
- agent 可能在同一 session 里隔三差五走一下 `fcop_report` 检查状态，但不为某份具体任务开「工作区」。

结果：tripwire 原设计上覆盖「开工作区」与「写报告」两个口子，覆盖不到「不开工作区，直接 StrReplace + git commit」路径。

## 提议方向（0.6.7 候选）

候选 A【软侧】：`fcop_report`（initialised）增加一项「工作区 / 任务单健康检查」——扫 `docs/agents/tasks/` 与 `git diff main` 后对比。若 `git diff` 非空、但 `tasks/` 里没有与之匹配的 OPEN 任务单，报告里增加一段明显警告：「WARNING: working tree has X uncommitted changes but no matching open TASK-*.md.」仍不阻塞、但警告明显。

候选 B【代价多】：提供一个 `fcop_audit_session` 工具（可选调用），调用后返回 session 期间未归档 task 与未完成 report。需要抽象出「session 总结。」

候选 C【代价低】：仅调整 `LETTER-TO-ADMIN.md` 中的「0.6.5 polish」表述，明确提示 tripwire 仅覆盖 `new_workspace` / `fcop_report` 两个入口；手工 git 的 agent 仍需自律。

推荐 A：实现量小、漏洞覆盖面限定在手工路径上、不引入新入口、不优先于现有 fcop_report 表现。
