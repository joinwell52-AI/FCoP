## 背景

v1 已按 TASK-007 验收、REPORT-009 归档，可作为一份合格的 "FCoP 协议演示稿" 发布。但 ADMIN 明确表示这份教程本来就是打算做成 **「免费开源·多 agent 实战教材·教你怎么指挥 agent」** 的。v1 偏协议术语，读者画像不够明确，没有抽象出可复用的“指挥 agent”方法论。必须以读者为中心重新调整。

## 三段心法（v2 的 spine，必须贯穿整本教程）

1. **单 agent 什么都做 → 记忆爆炸**。一个 session 什么都查、什么都记、什么都写、什么都改，上下文无限拉长，agent 开始忘事、自相矛盾、退化。多 agent 联合不是“炫技”，是解决记忆爆炸的必选路。
2. **MCP 工具是教你做 ADMIN 的**，不是教你写代码。我们不是码农，是指挥者、领导者；agent 是数字员工。学 fcop-mcp 不是学 26 个 tool，是学怎么当老板。
3. **每个文件落盘 = 管理 / 追溯 / 审计 / 有序 / 有规则**。把记忆从对话卸载到文件系统，谁说了什么、什么时间说的、为什么改了、为什么接了，都在文件里。filesystem is the protocol 不是口号，是涫营阳光下的可追溯。

## 加工方向（ADMIN 在 v1 验收后勾选）

### A · 读者画像 + 学习目标显式化

- 头部加三个框：**谁应该读 / 你会学到什么 / 你不会学到什么**。
  - 读者画像明确：在用 Cursor / Copilot 但快被单 agent 拖入记忆爆炸的开发者 / 独立开发者 / 技术 PM。**不是** FCoP 协议研究者；也**不是**问 "AI 会不会写代码" 那个层面的入门者。
  - “你会学到什么”列 3-5 条可检验能力：识别记忆爆炸的征兆、写出让 agent 落盘的 prompt、定义 ADMIN 边界、读 log 复盘、事故中收住边界。
  - “你不会学到什么”避免预期偏误：不会教写 JS、不会教调 LLM 参数、不会教训练模型。
- 每章末尾加一个“**本章你学到了什么**”小结 3 条，让读者能逐章自检。
- 顶部多加一句“**这份教程设计为让你从单 agent 跳上多 agent 协作、从 ”写代码的人“ 跳上 ”指挥 agent 的人“**”。

### D · 抽象为“指挥 agent 的 N 条铁律”（末尾专章）

在结尾“彩蛋”之后加一章【**指挥 agent 的 N 条铁律**】（N 推荐 5–7。不要超过 7）。要求：

- 每一条都是人话，不靠 FCoP 术语也能看懂。
- 每一条后面跟 "这在 fcop-mcp 里叫什么" / "换别的 MCP 工具怎么对应" 两句括号。
- 覆盖：
  1. 先识别再分配（不要一上手就 "你是 xxx"）
  2. 走四步循环（task → 实施 → report → archive，永远不要跳步）
  3. 一个 session 只演一个角色，别串戏
  4. 不删过往，只追加（你需要老账本，不需要漂亮的账本）
  5. 落盘是写给未来的自己看的，不是写给 agent 看的
  6. 越界发生时立刻归档成 ISSUE，不要掩盖（以第 4 章事故为案例）
  7. 记忆爆炸的信号：agent 开始重复问 、忘记忆之前的决定、自相矛盾、接不上刚才说的话——这时候不要靠 “提醒他记得”，要靠 “卸载到文件里 + 接手出去”
- N 你自己拍板；推荐 6 条。要静，不要为凑数加凑。
- 每条别超过 3 句，住手。
