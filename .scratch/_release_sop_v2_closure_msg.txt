docs(release): adopt v1.6.0 one-pass dry-run as canonical release SOP v2

Per ADMIN directive after v1.6.0 closure ("以后发版按这个来"), this commit
distills the actually-executed v1.6.0 release-pipeline experience back into
docs/release-process.md, plus closes the Rule 0.a.1 carrier
TASK-20260513-001 / REPORT-20260513-001.

What changed in docs/release-process.md
---------------------------------------

Rewritten from 259 lines (still using 0.6.0rc1 era examples) to 429 lines
covering the actual 1.x SemVer pipeline. Nine concrete gaps between the
old SOP and the v1.6.0 reality were absorbed:

  1. Version examples lifted from 0.6.x to 1.x throughout
  2. mcp/pyproject.toml dependency constraint now documented as
     MAJOR-lock (fcop>=1.0,<2.0), aligned with 1.x SemVer
  3. Lint/mypy release-gate scope explicitly bounded to
     src/ + tests/ + mcp/src/; scripts/ legacy lint marked non-blocking
  4. New section E. P=E sync (last step before tagging), with two
     paths: redeploy_rules() MCP tool (primary) and Project(...).
     deploy_protocol_rules() Python API (fallback when MCP server
     runs an older wheel — discovered live during v1.6.0)
  5. New documentation drift checklist under C. Docs covering README
     "Current release" line, letter-to-admin.{zh,en}.md summary
     block, plus 3 hard-coded test/server.py version-string asserts
     that bite if missed (all 3 tripped during v1.6.0)
  6. New section F. Rule 0.a.1 carrier as mandatory release
     deliverable: TASK-*-release-x-y-z.md + REPORT-*.md + archive,
     promoted from "nice to have" to a Core Invariant
  7. "Common failure 3: MCP server runs old wheel" added with the
     Python-API workaround command verbatim
  8. PyPI CDN propagation delay documented in Phase 4 + Common
     failure 6 (PyPI JSON API visible faster than pip index;
     observed ~5 min in v1.6.0)
  9. GitHub Actions Node 20 deprecation flagged (must fix before
     2026-06-02) in Common failure 5 + Automation roadmap

Also: an end-to-end command sequence section was added, so the next
release can be executed by following the SOP top-to-bottom without
needing the previous releaser's memory.
