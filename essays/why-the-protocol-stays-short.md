## 1 · 一个让协议维护者夜里睡不着的问题

如果你做过协议设计——任何协议,不论是网络、IPC、还是 agent 协作——你迟早会碰到这种夜里睡不着的时刻:**用户用了你的协议,而且越用越多,但每次他们都会做出协议没写过的事**。

第一次你会很兴奋,因为这种"涌现"证明协议**有用** —— 用户真的把它用起来了,真的拿它解决问题了,真的拿它去到了你没设想过的地方。

但第二次、第十次、第一百次之后,你开始焦虑。每一次涌现都需要回答同一个问题:

> 这件事我要不要把它加进协议?

加进去,协议变长一点;不加进去,下一个用户撞同一个问题,还得自己想一遍。两条路都不好。

更可怕的是第三种感觉:**这种涌现会不会没有止境?**

如果涌现真的没有止境、而协议又必须吸收每一次涌现,那协议会膨胀到没人能学得动 —— 学不动 = 失效。这是协议设计**最深的失败模式** —— 不是被竞品打败,不是被需求颠覆,而是**被自己的成功压垮**。

2026-05-12 的 codeflow 现场,一天产出 14 个涌现,达到了 FCoP 历史上的最高密度。这是一个**预演** —— 如果 FCoP 长期保持这个节奏,协议骨架会不会撑不住?

这篇就回答这个问题。

## 2 · 涌现的四种,只有一种该进协议

回答之前先做一件事:**把涌现分类**。

不是所有涌现都该被吸收进协议。如果不区分,协议方就会陷入"每一次涌现都焦虑"的状态,因为它默认 100% 的涌现都需要处理。事实上 100% 里**只有一小部分**该进协议,剩下的有更合适的归宿。

FCoP 维护者在实践中沉淀出的四象限分类:

| 类型 | 含义 | 进哪里 |
|---|---|---|
| **普适** | 影响所有项目 / 所有 agent 的涌现 | 进 `rules` / `protocol` commentary / 工具行为 |
| **团队特定** | 只对某类团队(`dev` / `media` / `mvp`)有意义的涌现 | 进团队模板,不进 rules |
| **项目特定** | 只对某个 codebase / 业务有意义的涌现 | 进项目本地 `fcop/shared/RULES-*.md`,不动协议 |
| **一次性** | 这次碰巧撞到,下次未必再来 | 进 essays / RETRO 历史,不入任何规则文件 |

举几个真实例子(均来自 FCoP 历史现场):

- **普适**:`supersedes:` frontmatter 字段(OPS 在 codeflow 现场发明) → 任何 FCoP 项目都可能需要,**进协议字段层**(`fcop_protocol_version` 2.0.0 → 2.1.0)。
- **团队特定**:`media-team` 的 `PUBLISHER` 角色需要的 `INDEX-{batch}.md` 索引文件(同一 deliverable 分包到 6 个 worker + 2 个 editor 时用)→ 只对内容生产团队有意义,**进团队模板**而不是 rules。
- **项目特定**:codeflow PM 的 "PowerShell encoding awareness" 自约束 → 对 Linux-only 团队是噪音,**进项目本地 RULES**。
- **一次性**:某次 OPS 在 commit 里 5 件 untracked 全 SHOULD-SKIP 的具体判断逻辑 → 下次未必同样组合,**留在 REPORT 历史**就够,不需要协议化。

只有第一行进 `rules` / `protocol`。其他三行**绝大多数**走更轻的归宿。

这就是协议方的第一条防线 —— **不焦虑,因为大部分涌现根本不该进协议**。

## 3 · 三条结构性力学:协议不会被涌现压垮

光分类还不够,因为分类只是把涌现**按归宿分流**。真正让协议骨架**不膨胀**的,是三条**结构性力学**。它们不是协议方主观努力的结果,是协议本身的形状决定的。

### 3.1 涌现密度天然下降

第一次有人用 FCoP,会撞到很多"协议没写"的事 —— 因为协议骨架还没补完,blind spot 大。
第十次有人用 FCoP,会撞到一些"协议写了但没说清"的事 —— blind spot 缩小,出现的是**理解 gap**。
第一百次有人用 FCoP,会撞到的是"协议写清了但具体 pattern 没列过"的事 —— 这一类涌现的**单次价值在下降**,通常进 essays / case studies 就够,不必进 rules。

涌现密度的下降是**对数的** —— 早期补骨架(每一条都填一块大空白),后期补案例(每一条都是已知 pattern 的小变种)。这条曲线是数学决定的,不是 FCoP 维护者拼命才换来的。

### 3.2 协议有自然边界

每一份好协议都有它的**自然边界**。HTTP 不管"你的 web 应用业务逻辑",TCP 不管"你的应用层语义",ETH 不管"你的链上业务对不对"。这是协议**留得住**的根本 —— 它只管它该管的事。

FCoP 的自然边界写在 `fcop-rules.mdc` `Scope` 节里:

> FCoP 只规范**协作协议**。它**不规定**:用什么模型、写什么代码、选什么框架、谁的 KPI、商业决策。

每次涌现来了,第一道闸是问:**这件事在 FCoP 边界内吗?**

绝大多数 agent 现场涌现是**协议边界外**的事 —— "这段代码该不该重构 / 用什么模型 / 业务规则对不对 / 团队怎么沟通"。这些事 FCoP 永远不接收,无论涌现密度多高。

闸门关掉后,真正进入"该不该收回"评估的涌现,大概只有原始涌现的 10%-20%。这是协议方的第二道防线。

### 3.3 每次收回都比上次更贵

协议每多一条规则 = 所有未来 agent 学习成本 +1。这是一个**严格的负反馈机制**:

- 协议方在收回每条涌现时,都会本能算"加进去后,所有未来 agent 都要学这一条"的隐性成本;
- 凡是"加进去不如写 essay 给后来者参考"的,自动走 essay;
- 凡是"加进去不如让工具自己处理"的,自动走工具行为;
- 凡是"加进去不如在团队模板里说"的,自动走团队模板;

—— 只有那些**真的没有更轻归宿**的涌现,才会进 rules。这条负反馈不需要协议方刻意维持,它**是协议方学习曲线的一部分**:协议方维护越久,这个本能越强,加进 rules 的门槛越高。

三条力学合起来,就是为什么协议骨架能在被持续涌现冲击下**保持稳定**。这不是设计师的克制,这是系统的结构。

## 4 · "最后"长什么样

把三条力学的结论外推到长期,会得到一张**多层增长曲线图**:

```
Rules / Protocol commentary    ── 增长曲线趋近水平
                                  Rule 0-9 + 当前 commentary 节是骨架
                                  v1.x → v2.x 可能再补 1-2 节
                                  v2.x 之后基本只动 patch / 措辞
                                  字数有上限 (~10k 中文字 + 等量英文)

ADR / Spec                     ── 缓慢、目的性增长
                                  每个重大决策一份 (~每月 1-2 份)
                                  累计上百份后增速会下降
                                  字数 ≈ 中速线性增长

Essays / Case studies / RETRO  ── 持续增长,无上限
                                  这是涌现真正的归宿
                                  每次现场都贡献 1-N 篇
                                  字数 ≈ 长尾,跟项目数线性

Drawer + .fcop/proposals/      ── 持续增长,无上限,git-ignored
                                  噪音留在私域,不污染历史
                                  字数 ≈ 无统计
```

四层增长速度**完全不同**:rules / protocol 接近冻结,ADR 缓慢爬升,essays / RETRO 长尾爆炸,drawer / proposals 噪音永生。

**这是分层的力量**。

新 agent 入场只学**最上层**(10 条 rules + 几节 commentary)就能干活,基本完整覆盖日常协作 —— 因为日常协作里**99% 的事**都被这层骨架处理完了。
当 agent 撞到协议没明示的事,**往下查一层** —— 先看 case studies / essays,有 80% 概率撞过的事别人写过了。
极少数情况要追溯到 ADR / proposals —— 这是"为什么协议是这样设计的"的考古层,新 agent 不强制学,但**留着对追溯有用**。

最上层短,下层无限长。这是协议**抗压**的根本机制。

## 5 · 类比:RFC 791 是 1981 年的

如果你觉得"协议骨架基本冻结"听起来不可信,看一眼网络栈的真实历史:

- **RFC 791**(IPv4 协议规范)发布于 **1981 年 9 月**。整篇规范大约 49 页。
- 截至今天,**40+ 年**过去了。
- IPv4 协议本身基本**没变** —— TTL 段定义、checksum 算法、fragmentation 规则,你今天读 RFC 791 跟 1981 年读 RFC 791 一字不差。
- 同期 IETF 发布的与 IP 相关的 RFC: **数千份**。覆盖 IPv6、TCP、UDP、ICMP、各种应用层协议、各种实践指南、各种性能调优经验。

40 年的网络演化**没有**让 RFC 791 膨胀。所有演化都发生在**它之上**的应用层、它周围的运维实践、它配套的案例集。RFC 791 本身**不动才是它能扛 40 年的原因** —— 一旦它动了,世界上几十亿设备都要跟着升级,生态会崩。

FCoP 想要的"最后"是同一种形状:**rules 文件冻结,周围的所有东西继续生长**。这不是空想,这是一个已经验证过的工业模式。

而且 FCoP 的优势比 IPv4 还大一点 —— **rules 文件升级时不需要升级几十亿设备**,只需要升级 `pip install fcop` + 运行一次 `redeploy_rules()`。但即便如此,**升级 rules 的代价仍然是真实的**(所有项目要重新部署 + 学新规则),所以 FCoP 维护者对待 rules 升级的姿态应该和 IETF 对待 RFC 791 升级一样:**能不动就不动**。

## 6 · 写给协议维护者的三条心法

如果你正在维护或者计划维护一份协议,而且你的用户每天都在产出涌现,以下三条心法你迟早会用到。

### 心法 1 · 涌现来了,先问归宿,再问要不要吸收

不要陷入"看到涌现就想加规则"的反射。先按四象限分类,大概率它根本不该进 rules。

**判断流程**:

1. 这件事在 FCoP `Scope` 边界内吗?边界外 → 拒绝,告诉用户去他自己的项目规则里写。
2. 边界内,但只对某类团队 / 某个项目 / 这次特定情境有意义吗? → 走团队模板 / 项目本地 RULES / essays。
3. 真普适,且没有更轻的归宿? → 才考虑进 rules / protocol commentary。

走到第 3 步的涌现,通常占原始涌现的 5%-15%。

### 心法 2 · 拒绝得起涌现,协议才活得久

每一条进入协议的规则都是**永久成本**。永久 = 所有未来 agent + 所有未来项目 + 所有未来扩展都要承担。

如果一条规则的**永久成本**大于它带来的**永久收益**(被多少人多少次用到),它**不该进协议**,无论它当下看起来多有道理。

这意味着协议方必须有**拒绝涌现的勇气**。当用户兴致勃勃地说"我发明了一个新模式,你看要不要加进协议",协议方应该有底气说:**"这模式很好,我们建议写进你的 `fcop/shared/RULES-*.md`,或者写一份 essay。如果未来三个月再有三个团队独立发明同一模式,我们再考虑收回。"**

这种"延迟收回"是健康的,因为它给了**社区验证期**。涌现要进协议,**至少应该证明它是稳定的**——一次发明不算稳定,N 个独立发明才算。

### 心法 3 · essays 不是次品文档,是协议**抗压**的核心机制

很多协议维护者把 essays / case studies / blog 当成"次要内容",觉得"真正的协议"在 spec / rules / RFC 里。这是错的。

essays 是协议**抗压**的核心机制 —— 它**吸收第三、第四类涌现**(让它们不进协议但也不丢失),它**为第一类涌现存档语境**(规则讲 what,essay 讲 why)。

没有 essays 这一层,所有涌现都会涌向 rules,rules 会迅速膨胀,膨胀后崩溃 —— 这是没有"涌现泄压阀"的协议必然结局。

essays 给协议**一个体面的"无规则承载层"**。涌现写进 essay,既保留了**叙事价值**(后来者可以读),又**没有变成强制规则**(后来者不必学)。两者兼得。

如果你正在设计一份协议,**从第一天就给它建一个 `essays/` 目录**。这件事的重要程度,和给协议建 `rules/` 目录一样高。

## 7 · 结 · 协议短,历史长

回到 ADMIN 那个原问题:

> 这样的涌现,是不是没有止境?最后呢?

**没有止境是真的**。FCoP 还会经历无数次 agent 撞墙、无数次野外发明、无数次"协议没写过"的情境。这是一份协议**活着**的证据 —— 死协议没有涌现。

**但协议不会被压垮**。三条结构性力学(密度下降、自然边界、负反馈)合力把绝大多数涌现挡在 rules 之外,挡下来的涌现按四象限分流到团队模板 / 项目本地 / essays / drawer,最终只有一小部分进入 rules,而那部分进入的速度还在持续下降。

最终的画面是:

- `fcop-rules.mdc` 字数趋近一个上限(~10k 中文字 + 等量英文),进入"40 年不动"的 stable 状态;
- `fcop-protocol.mdc` 跟随,接近上限后只做 patch / 案例引用增补;
- `essays/` 长尾爆炸,每一次野外现场都贡献 1-N 篇,几年后会变成 FCoP 最厚的目录;
- `.fcop/proposals/` 私域噪音永生,git-ignored,不污染任何人。

这就是 "**最后**" 长什么样:**协议短,历史长**。

如果说 RFC 791 是 1981 年定的、扛了 40 年,那 FCoP 的目标就是 —— **2026 年 v1.x 定下来的 Rule 0-9,要扛到 2050 年还能用**。中间无论发生什么涌现,rules 不动,历史在 essays 和 ADR 里堆积,协议骨架照旧。

涌现没有止境。协议不需要止境。两件事可以同时成立,只要分层做对。
