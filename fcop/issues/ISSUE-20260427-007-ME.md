## 实例场景

0.7.1 发版后 dogfood `fcop_report()`：

```
[版本 / Versions]
  fcop-mcp:  0.7.1
  fcop:      0.7.1
  rules:     （项目本地未部署）| 包内 1.7.0    ← ❌ 应是 1.8.0
  protocol:  （项目本地未部署）| 包内 1.6.0    ← ✅ 对
```

读 `src/fcop/rules/_data/fcop-rules.mdc`：

- 第 4 行 frontmatter：`fcop_rules_version: 1.7.0`（错）
- 第 560 行 body：`**Version**: fcop_rules_version: 1.8.0（见 frontmatter）`
- 第 563 行 body：`**1.8.0 changes / 1.8.0 变更**:` + 三条变更列举
- 规则正文已含 1.8.0 全部内容（sub-agent / Rule 5 / Rule 0.a.1）

即：**实质内容是 1.8.0，元数据是 1.7.0**，两者矛盾。

随 0.7.1 wheel 已发到 PyPI，新装用户跑 `fcop_report()` 看到的版本
号是 1.7.0（误导），但 redeploy 出来的规则文件行为是 1.8.0。

## 根因

**第三次"多行编辑只编了 N-1 处"同类 bug**：

| # | 何时 | 漏在哪里 | 怎么发现 |
|---|---|---|---|
| ISSUE-006 | 0.7.0 发版 | `mcp/pyproject.toml` 编了 description / version 字段，漏了 dep pin | uvx 用户 ImportError |
| REPORT-005 修订 | 0.7.1 收尾 | 决策从"yank"改成"不 yank"时漏改 4 处 doc | dogfood 复查 |
| **本 issue** | 0.7.1 发版 | 编了 rules body changelog，漏了 frontmatter 顶部 version 字段 | dogfood `fcop_report()` |

人眼审多行 diff 抓不住"N 处该一起改的有 1 处没改"——必须自动化。

## 影响

- **协议正确性**：无影响（实质规则内容是 1.8.0）。
- **元数据正确性**：错。`fcop_report()` / `redeploy_rules` 给出
  1.7.0，与实际行为不符。
- **用户体感**：以为协议没升级，可能不调 `redeploy_rules` 错过新规则。
- **同类风险**：下次再有版本字段散落多处的修改，没有自动化就还会
  漏。

## 0.7.2 决策

走 docs/metadata patch 0.7.2：

1. 修 `fcop-rules.mdc` frontmatter 1.7.0 → 1.8.0。
2. bump fcop / fcop-mcp 0.7.1 → 0.7.2。
3. **新增两道一致性测试**（核心防御）：
   - `test_rules_metadata_consistency.py`：frontmatter version 必等
     body 最新 changelog 标题号。
   - `test_package_version_consistency.py`：`_version.__version__`
     必等同包 `pyproject.toml` `version` 字段。
4. 0.7.1 不 yank（自动覆盖；同 0.7.0 决策路径）。
5. ADR-0002 如已含发版前检查表，再加一条"frontmatter version =
   body changelog 顶项"的核对动作。
