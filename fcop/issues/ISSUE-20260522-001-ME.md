## TL;DR

FCoP 3.0.0 / 3.0.1 已发布并在 spec/规则集层面把 `_lifecycle/{inbox,active,review,done,archive}/` 列为 **MUST**（`spec/fcop-3.0-spec.zh.md` §1.1），但**参考实现的初始化路径完全没有接通 v3**。任何全新项目跑 `Project.init_solo() / init_project() / create_custom_team()` 之后**都不会**得到 `_lifecycle/`，违反 spec MUST。

这是 P0 / blocker：协议层和实现层撒谎，文档（包括我刚改完的 `LETTER-TO-ADMIN.md`）跟着撒谎。

## 证据 / Evidence

### 1. `_apply_init` 只创建 v2 桶（src/fcop/project.py:924-931）

```python
for directory in (
    self.tasks_dir,
    self.reports_dir,
    self.issues_dir,
    self.shared_dir,
    self.log_dir,
):
    directory.mkdir(parents=True, exist_ok=True)
```

- 没有任何分支调 `fcop.lifecycle.state.ensure_lifecycle_dirs()`。
- 没有任何分支创建 `_lifecycle/{inbox,active,review,done,archive}/`。
- `init_solo` / `init_project` / `create_custom_team` 都收敛到这个 `_apply_init`，所以三条入口全部缺失 v3 落盘。

### 2. 拓扑探测形成自锁（src/fcop/project.py:240-248 + 332）

- `is_v3` 由 `_topology = detect_topology(...)` 决定。
- 全新项目里 `_lifecycle/` 不存在 → `topology="empty"` 或 `"v2"` → `is_v3=False`。
- `tasks_dir` property（line 331-333）此时返回 `<workspace>/tasks/`，不是 `_lifecycle/inbox/`。
- 即便 `_apply_init` 想"按 v3 落盘"，靠迭代 `self.tasks_dir` 也只会落 v2——属性不会主动切到 v3，除非 `_lifecycle/` 已经先于探测存在。**鸡生蛋问题**。

### 3. `migrate --to-v3` 救不了全新项目（src/fcop/cli/migrate_v3.py:292 注释 + REPORT-003 实测）

- `migrate` 设计为**对已有 v2 项目做迁移**。
- 全新项目里 `tasks/` 等是空的 → 探测判 `EMPTY` → migrate 直接 no-op。
- migrate_v3.py 自己注释里也写了：`A fresh init would create a v3 layout directly`——**表明开发者意图就是 init 直接落 v3**，但代码侧没接上。

### 4. spec 与 LETTER 互相打脸

- `spec/fcop-3.0-spec.zh.md` §1.1：v3 项目 **MUST** 维护 `_lifecycle/{inbox,active,review,done,archive}/`。
- 我刚替你改完的 `fcop/LETTER-TO-ADMIN.md` 也声明 "v3.0 标准拓扑" 是 init 之后的形态——这是基于 spec 的合理预期，但**当前实现不提供这个形态**，所以 LETTER 的这一段在事实层面**说了假话**（Rule 0.c 风险）。

### 5. 本仓自身也是 v2 状态

- `D:\FCoP\fcop\fcop.json` 的 `created_at = 2026-04-27T13:04:04`，早于 3.0.0 发布日（2026-05-21）。
- `D:\FCoP\fcop\` 下没有 `_lifecycle/` 目录（用 `Get-ChildItem` 实测过）。
- 这是 v2 项目，符合"未迁移"现状；但**新用户 / 新项目用 3.0.1 初始化也会落到同样的 v2 状态**，而非 spec 承诺的 v3——这才是真正的协议层 bug。

## 影响范围 / Impact

1. **协议合规** —— 全网所有跑 fcop 3.0.x 全新初始化的项目都不合规（未创建 spec MUST 的 `_lifecycle/`）。
2. **工具链表现** —— 写入路径（write_task / archive_task 等）通过 `tasks_dir` property 自动适配 v2/v3，本身"正确"，但因为初始化没建 v3 拓扑，整个 v3 写路径在新项目里**永远跑不到**。
3. **审计盲点** —— `Project.audit()` / `_scan_*` 系列**目前没有任何一项检查"v3 项目缺 `_lifecycle/`"**（grep `_lifecycle` 在 audit/scan_* 区域 0 命中）。这意味着 `fcop_audit` 不会把这个违规报出来——使用方完全感知不到自己处在违规态。
4. **文档** —— `fcop/LETTER-TO-ADMIN.md`、`README.md`、`docs/getting-started.md` 等全部需要根据"实际行为"重写；要么先承认 "3.0.1 init 仍然落 v2，必须显式 migrate 才能上 v3"，要么先把代码修好再让文档说真话。

## 建议处置 / Recommended Disposition

1. **代码侧（patch 3.0.2）** —— 在 `_apply_init` 末尾按 `version` 字段（或 ADMIN 选择）调 `ensure_lifecycle_dirs()` 落 v3 拓扑；或新增 `init_*(version="v3")` 显式参数让 ADMIN 选；或保留 v2 默认但提供 `init_solo(topology="v3")` 的明示开关。具体策略由 ADMIN 决定。
2. **审计侧** —— `audit()` 增 `_scan_lifecycle_topology_compliance()`：v3 项目（`fcop.json.protocol_version >= 3.0`）若缺 `_lifecycle/` 任何子桶，报 P0 violation 并指向 `migrate --to-v3` / 修代码后重 init。
3. **文档侧** —— 先按代码真实行为更新 LETTER / README，避免再次撒谎；代码修好后再回写 "v3 拓扑由 init 自动落盘"。
4. **MCP tools 一致性盘查** —— 35 份 tools/*.json 都是协议表层，只要 fcop Python 库的 Project facade 行为对，工具就跟着对；但 `init_solo` / `init_project` / `create_custom_team` 三条工具的 description 也得说明"3.0.x 当前默认仍落 v2，需 migrate"或在修复后写明"自动落 v3"。
