# ISSUE-011 · SOP §C 三处硬断言中两处版本漂移

## 现象 / Symptom

`docs/release-process.md` §C 明确规定:发版必查 **3 处 hardcoded 字符串**
是否与新版本号同步。本次 v2.0.0 release sprint 的 dry-run(D1)发现两处
仍停在 `"v1.6.0 摘要"`:

1. `mcp/src/fcop_mcp/server.py:779` —— `init_*` 工具回执 prompt 文本
2. `tests/test_fcop_mcp/test_server.py:163, 182, 196` —— 三处 `assert`
   断言(`test_init_project_reply_includes_letter_handover`、
   `test_init_solo_reply_includes_letter_handover`、
   `test_create_custom_team_reply_includes_letter_handover`)

`letter-to-admin.{zh,en}.md` 已升 `v2.0.0 摘要`(`tests/test_fcop/test_rules.py`
也同步了),但上述两处沿用旧字符串。`mcp/tests/` 70 passed 是因为 server
本身把"v1.6.0 摘要"写进 prompt 文本,断言查的就是那一份 hardcode,**两边
互相对齐着错了下去**——典型的 SOP §C 警告的"3 处硬断言"漂移。

## 影响 / Impact

P2 cosmetic:不阻塞 release,但 `init_*` 工具输出会告诉 agent
"念出 v1.6.0 摘要那一段",而 LETTER-TO-ADMIN.md 实际只有 v2.0.0 摘要
——agent / ADMIN 会困惑找不到对应段落。

## 根因 / Root Cause

C9 阶段(`_version.py × 2` bump、AGENTS.md / CLAUDE.md 重生成、
letter-to-admin 摘要标题升至 v2.0.0)忽视了 `mcp/src/fcop_mcp/server.py`
里 init_* 回执 prompt 的字面量,以及对应的 3 个 mcp/test_server 断言。
这正是 SOP §C 把这 3 处单独列出的原因——只升 letter 不升 server prompt
+ 断言,测试照样绿,但用户视角是错的。

## 修复 / Fix

`v1.6.0 摘要` → `v2.0.0 摘要`:

- `mcp/src/fcop_mcp/server.py:779` (1 处)
- `tests/test_fcop_mcp/test_server.py:163, 182, 196` (3 处)

修复后 `grep -r "v1.6.0 摘要"` 全仓零结果。回归:`tests/test_fcop +
tests/test_fcop_mcp` 972 passed + 1 skipped(skip 是 v2 minor gap > 1 条件
跳过,符合预期),`ruff check src tests mcp/src` All checks passed,
`mypy src` 21 files Success。
