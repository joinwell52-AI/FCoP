## 实例场景

2026-04-27 ADMIN 提交一份外部 agent 写出的 `AMEND-20260427-011-REPORT-...`
文件供 dogfood 分析。读这份文件时发现：

- `fcop-rules.mdc`（1.7.0 及之前）Rule 5 写到："需要修正时：追加新文件
  （`-v2` 或 `AMEND-*`），让历史可追溯。"
- `fcop` 库的解析器只识别三个前缀：`TASK-` / `REPORT-` / `ISSUE-`
  （见 `src/fcop/protocol.py` 的 `TASK_FILENAME_RE` / `REPORT_FILENAME_RE`
  / `ISSUE_FILENAME_RE`）。
- 因此外部 agent 严格按 Rule 5 字面落出 `AMEND-*` 文件，对 `list_tasks`
  / `list_reports` / `archive_task` 都是**幽灵文件**——从工具视角不存在。
- 同理 `*-v2.md`：`TASK-FILENAME_RE` 只允许 `TASK-{date}-{seq}-{sender}-to-{recipient}.md`，
  附加 `-v2` 后缀直接解析失败。

## 根因

协议规则与协议实现长期不同步。Rule 5 这一段写于早期，沿用至今但解析器
始终未跟进。"文件即协议"=文件名能被工具解析才算入账；写出去解析器看不
见的文件，违背了协议的根公理。

## 已采纳的方向（0.7.1）

**Rule 5 文案重写**——移除 `AMEND-*` / `*-v2.md` 例子，改为统一推荐
"下一序号同前缀"做法：

> 需要修正时：追加新文件——再写一条同前缀（`TASK-` / `REPORT-` /
> `ISSUE-`）、**下一序号**的文件，在新文件正文里说明"修正自 TASK-{date}-{seq}-..."。
> **不要**自创 `AMEND-*` / `*-v2.md` 这类前缀——FCoP 解析器只识别
> `TASK-` / `REPORT-` / `ISSUE-` 三种前缀，其它前缀对工具是"幽灵文件"，
> 违反"文件即协议"原则。

为什么不"反过来"扩展解析器去支持 `AMEND-*`？

- 增解析器→增前缀→增工具维护面，但所有可表达性已存在：sequential
  filename 加正文链接已经可以表达"修正自 X"，再加前缀只是冗余。
- 协议越简洁越能被人/AI 同时读懂，前缀爆炸是协议腐烂的早期征兆。
- 解析器改动是破坏性变更，需要历史扫描+迁移；rules 改文案是无副作用。

`fcop-protocol.mdc` 1.6.0 在 v1.6 changelog 里附了 1 句历史注，提示老
仓库里偶现的 `AMEND-*` / `-v2.md` 是 1.8.0 之前 Rule 5 的写法残留。
